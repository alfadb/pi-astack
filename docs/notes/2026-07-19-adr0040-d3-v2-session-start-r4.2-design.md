# ADR0040 D3-v2 session_start R4.2 static contract + dynamic execution capsule design

本文仅冻结 R4.2 设计，不实现代码，不生成或改写 evidence、transition、settings、session、control、rollback、audit 或 `.abrain` 内容，不执行 production preview/execute/continue/adoption/rollback，不 commit 或 push。本文中的 phrase 仅是未来实现必须匹配的协议模板，不是当前执行请求、授权坐标或阶段授权；当前状态保持 `S2_NOT_AUTHORIZED`，`proposition.adr0040-p3-d3-v2-session-start` 与 aggregate P3 保持 `blocked / separate_authorization_required`。

## 1. 修订定位与 R4.1 失败根因

R4.1 是 immutable predecessor；其两会话分离、trusted pi v3 JSONL、bind-existing-only、retained settings-parent OFD lock、duplicate-key rejection、create-only intent/activation/receipt、fresh continue recovery、selected-zero、独立 rollback 三门、dynamic source-commit gate、Linux/procfd 边界和诚实 residual 均不被本文回写或弱化。R4.2 是不兼容的后继 revision，任何未来实现必须使用新的 schema/version/path binding，不能把 R4.1 committed artifacts 改名后当作 R4.2 artifacts。

R4.1 的直接失败根因是 committed execution-ready dossier 冻结了整份 live settings 的 `settings_pre` full identity/raw hash、由该完整 preimage 渲染出的 `settings_post_raw_sha256` 以及含 live-derived binding 的完整 desired settings。这样，任何与 D3-v2 key 无关的正常 settings 编辑都会让 committed dossier、committed preview 与执行时 live A 不相等；为了恢复“可执行”，又必须重写 committed dossier，实际上把动态系统状态误当成静态授权合同，并诱导人对一次完整 live postimage 作字节级审阅。该模型既阻塞无关配置演进，也把可变 pre/post identity 错放进 committed content closure。

R4.2 将其拆为 `static contract + dynamic execution capsule`。committed static dossier/contract 只冻结变换器、允许的输入谓词、desired v2 template、mutation footprint、冲突规则和静态 pins；执行时在fresh coordinate与retained lock下捕获实际A、计算exact B、生成commit token和一次性tuple，再由R4.2 staged primitive durable发布canonical I/p消费该coordinate。live pre/post full identity 不进入任何 committed static artifact；post dossier 只能由 persisted tuple 与 receipt 派生。

## 2. 不变量与术语

`v2 key` 指 settings 中 `ruleInjector.propositionLifecycleFreshnessD3V2SessionStartInjection`。`non-v2 projection` 指从 duplicate-key-rejecting JSON root 删除该 v2 key 后得到的完整 JSON value，包含 `ruleInjector` 的其它 key 和 root 的所有其它 key；其 identity 为 RFC8785-JCS 字节的 SHA-256，不等于原始 settings 字节 identity。

`A` 是锁内捕获且通过 `P(A)` 的实际 settings preimage，包含 exact raw bytes、完整 file identity、strict parsed value、allowed v2 prestate、allowed metadata assertions 和 `non_v2_jcs_hash(A)`。`T(A, token)` 是 transformer 在 A 的 parsed value 上只写 v2 key、填入 tokenized desired v2 subtree并以冻结格式渲染得到的 exact postimage。`B` 是本 operation 期望提交的 exact `T(A, commit_token)` bytes。`C` 仅表示当前 settings 严格 JSON 中 desired v2 subtree、`r4Binding.static_contract_hash` 与 `r4Binding.commit_token` 均与 tuple exact，而 current raw bytes 既不等于 A 也不等于 B；从现场事实不能推导 CAS A→B 曾发生，也不能推导 C 是由 B 漂移而来，因此 C 只能成为独立 `ambient_state_adoption` 的候选当前状态，不能成为普通 continue recovery 分支。

`full identity` 固定为 `{dev, ino, mode, uid, gid, nlink, size, mtime_sec, mtime_nsec, ctime_sec, ctime_nsec, raw_sha256}`。实现必须使用 Node `stat/lstat/fstat(...,{bigint:true})` 暴露的`mtimeNs`/`ctimeNs`整数 nanoseconds，分别按 `sec = ns / 1000000000n`、`nsec = ns % 1000000000n` 规范化；先拒绝负值，再仅在每个 sec/nsec 可无损转换为 non-negative safe integer时写入 JSON，并要求 nsec 位于 `0..999999999`。禁止读取 fractional `mtimeMs`/`ctimeMs`、乘除浮点 milliseconds、截断、舍入或把 Node fractional ms 当作 identity。所有 raw hash 是 exact bytes 的 SHA-256；所有 object self-hash 均删除且只删除自身 hash 字段后，对 RFC8785-JCS 结果作 SHA-256。持久 control object 是 exact RFC8785-JCS plus LF、mode `0600`、bounded regular file、NOFOLLOW、create-only。本文所有 strict JSON 输入必须是无 BOM 的 exact UTF-8 JSON text，拒绝 UTF-8 BOM、UTF-16/UTF-32、注释、trailing comma、首个 JSON value 后的非 whitespace 内容、duplicate key、lone surrogate与非 RFC8785 可规范化数值；不得依赖 parser 的宽松模式、自动编码探测或 comment stripping。

`coordinate_hash` 是 trusted authorization transcript verifier 对完整 coordinate 的既有 self-hash。`fresh` 表示该 coordinate 是 bound authorization transcript 中最新 exact standalone `role=user` message、年龄不超过 2 小时、通过完整 v3 JSONL/header/parent-chain/active-branch/file identity/prefix 校验，并且在要求的锁内检查点仍是相同 coordinate。每个可能写入的 invocation 必须从首次验证起保留 authorization transcript 的 `O_RDONLY|O_NOFOLLOW|O_CLOEXEC` fd、已验证 header 与 frozen prefix；后续检查从该 retained fd 重读并验证 prefix、active branch、tail、latest coordinate、fd/path dev/ino 和 applicable later-than relation，不能用先前缓存结果代替。settings-parent OFD 只协调遵守协议的 settings/control writer，不锁 authorization transcript，也不能替代 retained-fd reverify。

`C_src`仅表示initial phrase coordinate中persisted dynamic `source_commit`，canonical I出现后还必须逐字等于tuple/intent及其派生V/R/gate中的`source_commit`；裸`C`在全文只表示第2节定义的ambient settings state，绝不表示Git commit。`source_guard(C_src)`是不可跳过的live guard：从repository object database重验exact `HEAD^{commit} == C_src`，逐个重读六个committed input path并要求live bytes SHA-256 exact等于`C_src:<path>`对应blob bytes hash，同时逐个重读全部manifest closure path并要求live bytes SHA-256同时exact等于`C_src:<path>`对应blob bytes hash与对应manifest row的`raw_sha256`，并重验tracked、nonignored、无staged/unstaged/untracked shadow及closure dependency row一致。实现可在branch/operation entry完整验证后缓存按UTF-8 bytewise排序的path list、manifest row和从`C_src` object取得的expected hash，但不得缓存、复用或推断任何live stat/read/hash/HEAD结果。每个可能mutation的branch/operation一旦从phrase或persisted tuple确定`C_src`，其branch-specific入口第一项必须完整重验HEAD、六artifact与全部closure/dependencies；此后每个mutating syscall都适用第6节紧邻`source_guard`，任何漂移时该syscall不得执行，并按此前mutation accounting返回`ZERO_WRITE_HALT`、`NO_FURTHER_WRITE`或runtime `selected_zero_injection`。

`canonical pending` 仅指由第7节 R4.2 staged primitive 完整发布且 strict bytes/self-hash/schema/owner/mode/inode 均有效的 deterministic authority name。`unique temp`是同目录、同设备、invocation-scoped的非权威staging inode，不进入I/V/R authority状态；其存在进入bounded inventory并阻止ordinary execute/continue/adoption/runtime mutation，只有same-invocation cleanup、same-inode normal convergence或第4/7/10节专用fresh disposition可执行各自窄write set。`ZERO_WRITE_HALT`表示本invocation尚无任何filesystem mutation即因漂移停止；`NO_FURTHER_WRITE`表示本invocation已经执行至少一个列明的mkdir/create/write/link/unlink/rename/sync mutation（无论其parent durability序列是否完成）后发现漂移并停止。两者都只输出非持久read-only conflict/incident report，不自动重试。

## 3. Static dossier 与 static contract

R4.2 committed execution-ready artifact 是 static dossier，不捕获 live settings A，不渲染 live B，不记录 live pre/post raw hash、full identity、inode、mtime、ctime、size 或动态 authorization coordinate。static dossier 包含 closed `static_contract`、其 `static_contract_hash`、静态 assertions、content-only source closure result、committed preview template identity、`S2_NOT_AUTHORIZED` 状态和 dossier self-hash；其中只有 `static_contract_hash` 是执行 phrase、commit token、tuple 和 runtime binding 的静态根。六个committed inputs及其各自self-hash preimage都不得包含具体`source_commit`/`HEAD` OID；static dossier与static preview template只能携带closed policy常量`source_commit_binding="dynamic_preview_and_initial_phrase"`，不得携带OID，static contract的三个manifest projection同样无`source_commit`。

`static_contract_hash = SHA256(RFC8785-JCS(static_contract_hash_preimage))`，其中 `static_contract_hash_preimage` 必须从 committed static contract artifact 的 strict parsed value重建：shallow-copy整个object，删除且只删除top-level `static_contract_hash`，并原样保留nested `settings_contract.desired_v2_template.r4Binding.static_contract_hash`中的fixed closed sentinel `{"$static":"static_contract_hash","encoding":"lowercase_hex","length":64}`。Committed static contract artifact的top-level `static_contract_hash`必须回填上述computed 64位lowercase hex final hash，validator必须重算并逐字相等；“禁止回填”仅指nested `desired_v2_template` slot，该nested位置在artifact中永久保持sentinel，绝不能回填final hash string。因此hash preimage不依赖任何“先回填nested再裁剪”的自引用步骤。`static_contract` 除 `schema_version`、`revision`、`canonicalization` 与 top-level `static_contract_hash` 外的 hash 字段闭集仅为下表所列分组；validator 必须 exact-key reject，禁止 extension bag、live snapshot、preview result 或 future post identity 混入 hash preimage。`source_manifest`、`adapter_manifest`与`operator_manifest`三个字段只保存各自standalone manifest的closed immutable identity projection，不复用standalone manifest exact schema，不嵌入closure rows、schema pins、dependency pins或任何`raw_sha256`，也不对projection另作self-hash；每个projection字段必须从对应exact live standalone manifest及其冻结artifact path机械取得并逐字段相等。

`settings_contract.allowed_v2_prestate` 固定为 `absent` 或 exact `{"enabled":false,"selector":{"session_ids":[]}}`，且不得把其它 disabled shape、legacy activation pins、已有 R4、R4.1或R4.2 binding或已有target selector视为允许prestate。`settings_contract.allowed_metadata_policy`固定为NOFOLLOW regular file、`nlink==1`、`uid==effective_uid`、`gid==effective_gid`和mode closed set `[0600,0644]`（JSON整数分别为384、420）；实现不得根据caller或单次A动态扩展该closed set。A capture必须在retained settings-parent fd下同时比较fd/path identity并拒绝其它type、link count、owner、group或mode；tuple保留A full identity并写入closed metadata assertions。`transformer_version`必须是source-bound、schema-bound的immutable identifier；变换器版本变化必须产生新static contract hash。

`desired_v2_template` 冻结 exact v2 value：`enabled=true`、只含 target session 的 selector、D3 pins、adapter manifest hash、`maxReadBytes` 和 R4.2 binding template。Static hash slot的唯一表示是位于`r4Binding.static_contract_hash`的exact closed object `{"$static":"static_contract_hash","encoding":"lowercase_hex","length":64}`；dynamic token slot的唯一表示是位于`r4Binding.commit_token`的exact closed object `{"$dynamic":"commit_token","encoding":"lowercase_hex","length":64}`。两类slot各恰好出现一次且path固定，任何其它`$static`/`$dynamic` object、重复slot、错误path、额外key、默认值补全或执行时option均拒绝。Static contract artifact与committed preview template必须保留static sentinel和dynamic sentinel；渲染desired subtree/B时先把整个static sentinel替换为computed top-level `static_contract_hash` string，再按第5节late-bind整个commit-token sentinel。Persisted desired subtree、settings B、tuple、activation与receipt必须只含两个匹配`^[0-9a-f]{64}$`的final string并拒绝任一残留sentinel；反之static contract artifact若nested位置已是final hash string也必须拒绝。`r4Binding`其余字段固定schema version、control root、operator manifest hash和settings path。

`mutation_read_set`至少包含settings full raw/identity与allowed metadata policy/assertions、root object、`ruleInjector` object、v2 key、v1 selector、old activation root、target/auth bindings、source closure、D3/static pins、retained authorization transcript fd/prefix、完整bounded control inventory、settings-parent内匹配R4.2模板的bounded settings-temp inventory和rollback root absence。R4.1 legacy audit path pin仅是可选只读历史定位，不进入operator mutation predicate、readiness/conflict或R4.2 runtime terminal gate。`mutation_write_set` 的总闭集仅为 v2 key replacement、该 operation 的 intent/activation/receipt final与deterministic pending、各kind invocation-scoped non-authority unique temp、same-directory settings CAS unique temp及其向A mode/uid/gid的metadata初始化、必要的 exact control bootstrap prefix、专用staged-temp disposition允许的unlink与parent fsync及可选best-effort operator audit；R4.2 per-receipt runtime audit由第10节独立runtime合同控制，不属于operator write set，session、D3、old activation root、rollback root内容和其它settings key都不在其中。所有authority object必须使用第7节R4.2 staged primitive；temp始终是non-authority，绝不能被runtime、coordinate-consumption scan或I/V/R closure当作authority。Initial execute按第6节全序使用该闭集；`ambient_state_adoption`本身不写settings且其可自动变更入口必须按第8节进入A/B0/B/C/D五个互斥分支：A首次发布adoption R，B0从canonical I与static closure确定性渲染并发布此前absent的exact V，B只收敛唯一既存exact V/p，C只链接pending-only adoption R/p，D只移除同inode同bytes adoption R+R/p的pending link；B0/B稳定重验后才可转A，C转D必须重新分类并独立验证。ordinary continue对合法C candidate只输出内存/标准输出中的non-authoritative adoption preview，I/p绝不由adoption收敛。

Canonical intent pending/final是initial capability的首个authority write与唯一durable消费witness。Control bootstrap不是一次性“全有或全无”布局，而是固定单调non-authority prefix automaton：`B0=control root absent`→`B1=control root exists and empty`→`B2=B1+empty intents`→`B3=B2+empty activations`→`B4=B3+empty receipts`，且每个state都要求所有已存在目录为current effective uid/gid、mode exact `0700`、same filesystem device、regular directory、ancestor/self no-symlink，entry集合逐字等于该state且无`operator-audit.jsonl`、temp、pending、final或extra。只有全部second reverify成功后才能进入B0→B1；每个`mkdirat`与对应parent `fsync`都必须在完成所需state检查后执行第6节紧邻auth/target+`source_guard(C_src)` prelude并立即进入syscall。Crash、expiry或latest漂移可留下exact B1/B2/B3/B4；目录不消费coordinate。Later fresh/latest initial coordinate必须在新invocation重新捕获A、重算B/token/tuple/op并重验source/session/rollback absence与exact prefix state，先补做相应parent fsync，再按唯一下一目录幂等推进；达到B4后还须执行一次post-bootstrap bounded revalidation，成功才可进入intent staging。任何跳序（例如无`intents`先有`activations`）、extra entry、错误uid/gid/mode/dev/type、symlink或alias都halt且不修复。若A/auth/source/session drift在首个mkdir前发现则`ZERO_WRITE_HALT`；若bootstrap已推进后发现则保留safe prefix并`NO_FURTHER_WRITE`，但仍无coordinate consumption。Initial coordinate只在完整canonical I/p经hardlink no-replace、parent fsync与strict readback成功发布时durably consumed；adoption coordinate只在完整canonical adoption R/p以同样步骤首次成功发布时durably consumed。Temp的`O_EXCL` create、write、file sync或readback，目录bootstrap，V/p publication、B0/B收敛V、ordinary continue、temp disposition与recovery gate均不是durable consumption witness。因为compliant operator从不打开authority pending作增量写，deterministic pending只能absent或一次hardlink出现为完整canonical bytes；任何partial、zero、strict-unparseable或parseable-wrong deterministic pending均统一分类为`F/corrupt_external_authority_residue`，不能推导任何coordinate已消费，也不是normal crash state；这类corrupt authority仍是external incident，只能由本文范围外的独立协议处理。

`conflict_rules` 至少冻结：duplicate key、非-object root/ruleInjector、非 allowed v2 prestate、target出现在任何v1 selector或任一foreign/legacy v2 selector、任一旧R4、R4.1或R4.2 binding（包括disabled但带R4.1 binding的shape）或foreign static/token pin、foreign authorized activation、target/auth alias、control/rollback alias或nesting、rollback root在initial/pre-S2已存在、foreign/corrupt authority、未经第7节专用preview+phrase绑定的staged temp、source/session/auth drift、static pin mismatch、settings symlink/type/mode/link/owner/group violation、target binding drift、非法A/I/V/R组合全部fail closed。target session id/path/dev/ino/prefix任一漂移都必须halt，即使commit token公式本身不变；已构造tuple不得改绑target。冲突规则不能由CLI flag、preview result或caller payload放宽。

Static source clean closure由三个各自独立的standalone manifest承载。三者共同exact keys为`relative_path`,`schema_version`,`revision`,`kind`,`closure_rows`,`graph_hash`,`source_closure_hash`,`file_count`,`self_hash`；source另含`critical_required_paths`,`clean_closure_policy`，adapter另含`staged_publication_schema`,`link_final_schema`,`unlink_pending_schema`,`staged_temp_path_schema`,`dependency_pins`，operator另含`operator_entry`,`cli_entry`,`generator_entry`,`schema_entries`。三个standalone manifest及其self-hash preimage均无`source_commit`或其它具体Git OID字段，并禁止出现表示自身文件bytes的`raw_sha256`或任何等价self-raw-hash字段；`self_hash`只从strict object shallow-copy删除且只删除top-level `self_hash`后对RFC8785-JCS bytes计算，manifest文件exact raw SHA-256只能由外部validator在完整JCS+LF bytes生成后计算，并可记录于不进入manifest closure的static dossier `source_closure_result`。`closure_rows`按relative-path UTF-8 bytewise排序，row exact为`{relative_path,raw_sha256,dependencies}`，dependencies是sorted unique relative-path array，`file_count`必须exact等于rows长度；operator的entry projections必须逐项命中自身closure row并与其relative path/raw hash相等。属于adapter或operator图的row，其dependencies必须逐字等于对应静态依赖解析器得到的sorted unique direct dependencies；仅因`critical_required_paths`进入source closure且不在adapter/operator图中的critical-only row，也必须由冻结的static artifact generator解析出sorted unique direct dependencies，若解析结果无依赖则必须是exact empty array `[]`，不得省略、填null或把unknown视为空。`adapter_closure_set`必须覆盖R4.2 staged publication、anchored open/read/write/link/unlink/fsync、strict JSON/self-hash、retained transcript reverify及其全部transitive dependencies；`operator_closure_set`必须覆盖operator code、CLI、所有本节schemas、artifact ## 5. Commit token 与 dynamic execution capsule

`commit_token` 是 v2 subtree 与 persisted operation capsule 之间的不可循环绑定，不是 secret，也不是 authorization bearer。其 exact domain-separated 公式固定为：

```text
commit_token = SHA256(
  UTF8("adr0040-d3-v2-session-start-r4.2-commit-token/v1\0") ||
  ASCII(static_contract_hash) || NUL ||
  ASCII(coordinate_hash) || NUL ||
  pre_raw || NUL ||
  ASCII(source_commit) || NUL ||
  UTF8(target_session_id)
)
```

`pre_raw` 是 A 的 exact raw bytes，不是 raw hash、JCS bytes 或 parsed JSON；前置 domain separator、固定字段顺序和 NUL delimiters 是公式的一部分。settings JSON 不允许 NUL，因此边界无歧义。token 公式不含 operation id、expected B hash、receipt 或 post identity，避免自引用；知道 token 的 writer 仍不能绕过 mutating continue gate、专用 adoption gate、intent/activation closure或 create-only receipt publication。

协议中的`render exact B`步骤先验证static artifact内恰有两个fixed-path sentinel，然后从A与static desired template建立确定性的late-bound byte plan。第一步把`r4Binding.static_contract_hash`整个static sentinel替换为按第3/13节从artifact重建preimage得到的computed top-level hash string；此时不得改动commit-token sentinel。第二步按独立公式计算token，再把`r4Binding.commit_token`整个dynamic sentinel替换为fixed-width 64 lowercase-hex string，最后完成exact B bytes与`expected_B_raw_sha256`。实现可以在token计算后作等价的一次deterministic final render，但必须证明替换顺序与两个唯一path不变，且不得把provisional/final B hash输入token。Static contract validator必须从artifact sentinel重建hash preimage；static artifact中的nested final hash、persisted settings/tuple/V/R中的任一sentinel、零值占位token或长度非64表示均拒绝。

Dynamic execution capsule 的 persisted authority 是 intent 内的 closed `operation_tuple`。tuple 必须包含 `prestate_A.full_identity`与 decoded `prestate_A.raw_base64` exact bytes、allowed v2 prestate、`prestate_A.raw_sha256`、`prestate_A.non_v2_jcs_hash`、从A fd/path现场验证并与full identity逐字段闭合的`prestate_A.metadata_assertions`、expected B raw hash、exact desired v2 subtree（含 `r4Binding.static_contract_hash` 与 `commit_token`）、commit token、initial coordinate/full coordinate hash、由initial dynamic preview与exact phrase绑定并锁内复验的persisted dynamic `source_commit=C_src`、target/auth bindings、D3 pins、manifest/static contract pins、transformer version、mutation/conflict/path pins和 safety residual pins。tuple 不得包含未来 post full identity、receipt mode、ambient C、preview output、wall-clock retry state或 caller overrides。

tuple 中 A raw只保留一种字节表示：`prestate_A.raw_base64` 是 RFC4648 standard alphabet、带规范 padding且解码后再编码必须逐字相等的 string；`prestate_A.raw_bytes` 只是非负 safe integer byte count，不是第二份表示。validator 必须验证 decode length exact等于 `raw_bytes`、`full_identity.size`，decode SHA-256 exact等于 `raw_sha256`与 `full_identity.raw_sha256`，`metadata_assertions`的regular/nlink-1/effective uid/effective gid/allowed mode/fd-path identity/NOFOLLOW七项均为true且分别与`full_identity`及static `allowed_metadata_policy`机械一致，随后对 decoded bytes作 strict JSON/duplicate-key检查并重算 allowed prestate与 `non_v2_jcs_hash`。这里公式中的 `pre_raw` 是解码后的原始 bytes。tuple object继续使用 RFC8785-JCS；settings 与 tuple各自上限为1 MiB，超过上限在 intent前零写拒绝。

`operation_id = SHA256(RFC8785-JCS(operation_tuple))`。intent self-hash覆盖schema、operation id、顶层逐字等于tuple的persisted dynamic `source_commit=C_src`、完整tuple和resolved control paths；activation绑定operation id、intent hash、exact desired v2 subtree/token、target session、D3、source/static pins、runtime/rollback/audit paths，但不复制或发明post full identity。Receipt closed union仅有`direct`与`ambient_state_adoption`两种mode；direct只证明锁内观察到exact B，adoption只证明经专用授权采纳了当时exact C。Tuple、intent顶层、activation、receipt、continue/adoption/recovery/disposition phrase与gate、runtime enable/disposition preview与gate中的`source_commit`都必须逐字等于initial tuple持久化的dynamic `C_src`；后续live `HEAD^{commit} != C_src`或`source_guard(C_src)`任一六artifact/closure live hash检查失败时，所有control/bootstrap/authority/settings/audit/disposition/recovery/runtime mutation均拒绝，且不得重绑新HEAD。

## 6. 锁内全序、逐 syscall 授权重验与消费边界

Initial execute的唯一合法全序如下，箭头表示前一步完整成功后才能进入下一步；第一次auth读取和OFD获取本身不写文件：

```text
auth through retained transcript fd/prefix, exact preview transcript boundary and phrase-bound source_commit C_src
-> acquire retained settings-parent OFD lock
-> branch/operation entry first completes HEAD^{commit} == C_src + six artifacts + all closure rows/dependencies exact C_src:path/manifest revalidation
-> reverify same initial coordinate fresh/latest + post-preview later relation + target/auth bindings
-> capture A with duplicate-key rejection and full identity/raw
-> evaluate P(A)
-> render exact B plan, compute token bound to C_src, finalize B, tuple and operation_id
-> one bounded revalidation of exact A/auth/session/control inventory + rollback-root absence while source entry result remains current
-> bootstrap B0->B1->B2->B3->B4 by ordered mkdirat+parent fsync, each syscall immediately preceded by auth/target + source_guard(C_src)
-> one post-bootstrap bounded revalidation of exact A/auth/session/B4 inventory + rollback-root absence and complete source_guard(C_src)
-> R4.2 stage temp -> publish canonical I/p -> link I final -> unlink I/p
-> R4.2 stage temp -> publish canonical V/p -> link V final -> unlink V/p
-> recompute non_v2_jcs_hash(A/B), immediate/final exact A check
-> write+sync settings temp -> anchored rename A to B -> parent fsync -> exact B readback
-> R4.2 stage temp -> publish canonical direct R/p -> link R final -> unlink R/p
-> verify terminal closure
```

Intent前的pre-bootstrap与post-bootstrap两次bounded revalidation都要求A按full identity/raw重读相等，initial coordinate通过retained transcript fd仍为same post-preview applicable later/fresh/latest coordinate，dynamic `source_commit`仍逐字等于phrase/preview绑定的`C_src`，六artifact与closure rows/dependencies仍live exact `C_src:<path>`且clean，target/auth prefix/file identity相等，rollback root/stateRoot exact absent；前者要求control root为第3节B0..B4之一，后者要求exact B4。每个initial/continue/adoption/disposition/recovery/runtime branch或operation入口的第一项source contract必须在任何该branch mutation前完整重验HEAD、六artifact和全部closure/dependencies；preview `C_src1`→现场HEAD `C_src2`或artifact/closure drift在该边界一律`ZERO_WRITE_HALT`或runtime selected-zero并要求new applicable preview/phrase。B0到B4只能按固定顺序推进；每个mkdir及其parent fsync都必须执行下一段紧邻guard prelude，crash/expiry后later fresh coordinate只能在仍绑定同一`C_src`且entry full revalidation与逐syscall source guard完整通过时补全exact prefix，目录本身不消费coordinate。其它失败在首个目录/temp写入前返回`ZERO_WRITE_HALT`；bootstrap已经产生目录后才发生的auth/session/source drift则按诚实mutation accounting返回`NO_FURTHER_WRITE`并保留safe prefix，且同样要求new preview/phrase，不得把`C_src2`改绑进operation。Intent后禁止重捕获A形成新operation；V、settings CAS、R和任何convergence都只能从persisted I tuple解码A/B/token/static pins，不能重新运行transformer适应live值。Mutating continue先发现sole operation，再验证closed tuple、initial provenance与对应fresh continue/adoption/recovery coordinate，之后只能进入本节和第7节R4.2 primitive，不能直接调用或语义复用R4.1 publication implementation。

preview/phrase处理：path/metadata/raw descriptor必须绑定，temp mode/uid/gid必须exact等于A，live settings必须strict exact A且operation I/V closure有效，或strict exact B且该temp仅为冗余exact B inode；若settings为C/X、多个matching temp、wrong metadata/path/op或无法证明relation则fail closed。Disposition不执行rename、不推进CAS，完成后必须另取fresh continue。每个invocation对non-cooperative settings drift只允许一组back-to-back settings read与一次bounded closure revalidation，不循环重读直到相等；漂移立即bounded exit并要求人先停止或协调conflicting writer、重新获取read-only preview和fresh authorization后再调用。该外部并发是liveness residual，R4.2不保证eventual progress，但所有退出都保持runtime selected-zero或既有合法terminal，不产生不安全terminal。

## 7. R4.2 staged publication、幂等 convergence 与 incident 边界

Final authority paths固定为`intents/<operation_id>.json`、`activations/<operation_id>.json`和`receipts/<operation_id>.json`；deterministic pending固定为`.<operation_id>.<kind>.pending`；non-authority temp固定为`.<operation_id>.<kind>.stage.<invocation_nonce>.tmp`，其中kind依目录为`intent`、`activation`或`receipt`，nonce为该invocation以CSPRNG生成、进程内唯一且此前未出现在inventory的32位lowercase hex。R4.2 adapter必须分别实现schema-pinned `staged_publish/v1`、`link_final_idempotent/v1`和`unlink_pending_idempotent/v1`，不得import、dispatch或直接复用R4.1的pending publication/link/unlink实现；R4.1保持immutable。

`staged_publish/v1`按以下不可重排步骤工作。先执行第6节完整guard prelude并立即以anchored same-directory `openat(O_RDWR|O_CREAT|O_EXCL|O_NOFOLLOW|O_CLOEXEC,0600)`创建unique temp，并从fd与path同时验证regular file、`uid==effective_uid`、`gid==effective_gid`、mode exact `0600`、nlink-1、same parent device和retained dev/ino；temp create返回`EEXIST`表示pre-scan后的foreign race，必须halt并read-only incident，不能重选nonce后继续写；随后在对象上限内完整write exact JCS+LF，每次short-write retry前重新执行第6节完整guard prelude，并在`fdatasync(temp_fd)`与`fsync(temp_fd)`各自紧邻前分别执行同一完整prelude后立即进入syscall，再从retained temp fd和anchored path strict readback exact bytes/size/schema/self-hash/mode/owner/dev/ino/nlink-1。只有这些步骤全部成功后，才以anchored hardlink no-replace把该inode发布到deterministic pending；link前再次执行第6节完整guard prelude，成功后在`fsync` parent前再次执行同一prelude，再strict readback pending与temp同inode/同bytes/nlink-2。Temp create/write/sync/readback都不是authority publication；只有该pending link、parent sync与readback全部成功，canonical I/p或首次canonical adoption R/p才分别durably消费initial/adoption coordinate。V/p和direct R/p publication不消费其coordinate。

Temp→pending hardlink返回`EEXIST`时不得覆盖；同一invocation只能在重新扫描后将“pending exact canonical、与retained current temp fd同inode/同bytes、temp path仍为同inode且nlink-2”识别为该次调用的幂等成功，随后仍在fsync parent前执行第6节完整guard prelude并readback。其它`EEXIST`、`ENOENT`或metadata/bytes差异全部halt。Pending durable publication后，同一invocation必须在unlink与fsync parent各自紧邻前执行第6节完整guard prelude，并readback canonical pending nlink-1；unlink返回`ENOENT`仅在pending仍与retained temp fd同inode/bytes、temp absent、nlink-1时幂等成功，否则halt。若auth、binding或source在temp已写但pending发布前漂移，只允许同一invocation在再次通过第6节完整guard prelude后删除exact current nonce、same retained inode、non-authority nlink-1 temp并fsync/readback parent；检查无法通过则`NO_FURTHER_WRITE`并保留temp。Crash或invocation结束留下的well-formed sole-operation temp不自动成为authority或被execute auth删除：pending/final absent且temp nlink-1时进入read-only `staged_temp_disposition` candidate，由专用fresh phrase绑定后仅unlink；temp与canonical pending same inode/exact bytes/nlink-2时由该kind normal convergence gate先unlink redundant temp并fsync/readback，再继续pending convergence。Unknown name、wrong operation/path/device/owner/mode/link relation、multiple temp或与pending不同inode/bytes仍是F incident。

`link_final_idempotent/v1`只接收已strict验证且由retained pending fd绑定的canonical object，执行第6节完整guard prelude后立即执行pending→final hardlink no-replace。成功后在parent fsync紧邻前再次执行完整guard prelude，readback必须证明final/pending与retained object同inode、同bytes、nlink-2。`EEXIST`后必须重新分类，且只有两种结果可视为幂等成功：final+pending均与retained staged object同inode/同bytes且nlink-2；或final-only与retained staged object同receipt/object inode/bytes、pending absent且nlink-1。两种结果都必须在parent fsync前再次执行第6节完整guard prelude并strict readback；任何其它final、pending、inode、bytes、mode、link-count或receipt-mode组合立即halt。`unlink_pending_idempotent/v1`只在分支write set允许时执行anchored pending unlink；成功后在parent fsync前执行第6节完整guard prelude并readback final与retained object同inode/bytes nlink-1。`ENOENT`后仅在重新分类证明final为同receipt/object inode/bytes nlink-1且pending absent时幂等成功，随后仍须在parent fsync前执行第6节完整guard prelude并readback；否则halt。任何link/unlink race都不能通过“目标已存在/不存在”直接返回成功。

完整control inventory采用deterministic bounded scan：bootstrap state按B0..B4逐级检查；B4后control root最多4个top-level names，即exact三个authority directories加可选`operator-audit.jsonl`。每个authority directory最多3个entries，即该kind至多一个final、一个deterministic pending和一个unique temp，总计最多9个authority/staging names；settings parent另最多允许一个匹配sole operation冻结模板的settings CAS temp。枚举按UTF-8 bytewise ascending排序并确认不存在第10个control entry或第二个settings temp；超限、foreign name或multiple operation立即halt。Temp按互斥状态分类为`Tcur`（本invocation nonce+retained inode）、`Td`（prior-invocation、well-formed、sole-operation且metadata/relation可由read-only disposition preview闭合）或`F`；只有Tcur可在原operation gate内按current-invocation规则cleanup，Td必须使用第4节专用phrase，不能被execute/continue/adoption直接采用。Canonical pending/final按sole operation、exact path/bytes/schema/self-hash/mode/uid/gid/size/inode/nlink分类；partial、zero、unparseable、parseable-wrong deterministic authority pending始终为`F/corrupt_external_authority_residue`，但partial/wrong non-authority temp只要path/inode/metadata/inventory可精确绑定仍可为Td。

任何F只产生bounded non-authoritative `incident_report`，exact绑定`static_contract_hash`、control root、按bytewise顺序列出的完整bounded control/settings-temp inventory，以及每个residue的absolute path、dev、ino、raw SHA-256、size、mode、uid、gid、nlink和classification；不能安全读取raw bytes时hash字段使用closed reason union而不是伪造hash。R4.2 operator永不删除、quarantine、rename、overwrite或re-enable未知/foreign/corrupt authority；这仍属于本文范围外的独立manual protocol。Td则生成第4节closed non-authoritative `staged_temp_disposition_preview`，不是incident豁免或写授权；只有exact fresh standalone disposition phrase可调用R4.2内冻结的窄unlink入口。Compliant operator不会增量写deterministic pending，因而不能产生partial/zero/unparseable authority pending；corrupt authority F不是normal crash state，而是external residue incident。F entry本身不能证明coordinate consumption；若同一inventory另有strict canonical I/p或adoption R/p，其provenance/consumption语义不被temp抹除。Td在处置前仍阻止execute/continue/adoption/runtime mutation，处置后必须新fresh gate；同inode pending+temp nlink-2则只能由normal convergence gate处理。

Intent-only、intent+activation、CAS-before-receipt和final+pending窗口只按本节R4.2 primitive恢复。Continue的A路径要求current full identity/raw exact等于tuple A；B路径要求current raw bytes/hash exact等于tuple expected B且v2/token exact，并从tuple decoded A和current B重算non-v2 JCS相等；C路径不得direct recovery，只能按第8/12节在canonical I final及V absent/V/p/V+V/p/V final inventory中零写preview。Current exact B与I/p共存时ordinary mutating continue可按operation continue合同收敛I，再发布V、重验并direct完成；adoption看到I/p始终halt且不顺带收敛。Current exact B与canonical I final、V absent、R absent时continue使用staged primitive发布expected V并重验后才可发布direct R。Current C与canonical I final、V absent时ordinary continue输出含`activation_absent=true`与expected activation hash的preview，只有B0专用adoption权限可创建exact V；C+I/p仍halt且不输出preview。I发布后若current不再是exact A/B或合法C，operation不得重捕获A或改写tuple，只能halt并保留authority。

## 8. CAS→receipt crash、mode-first recovery 与独立 state adoption

CAS exact B readback后、direct receipt durable publication前存在crash窗口；external writer也可能在未知历史下种入保留desired v2/token的C。Token可读且C不能证明历史，ordinary continue不得把C解释为direct CAS recovery。Ordinary continue在sole canonical I final、R absent、V absent/V/p/V+V/p/V final之一、current C与static/source/session/auth closure有效时，只作一组back-to-back C双读和一次bounded revalidation，零写返回第4节含closed `activation_hash_inputs`及全部content/inventory bindings的`adoption_preview`；V absent时必须显式`activation_absent=true`并给出由independent validator从I/static重算的activation nonce、expected bytes与`expected_activation_object_hash`。I/p、F、Td或任何receipt存在时不得输出该preview。

Receipt mode具有分类优先级：发现任何R/p或R+R/p后，operator必须先在不写settings/control的情况下strict parse receipt exact schema、self-hash、mode、completion authorization、post witness、bytes/inode/closure，再按persisted `mode`唯一选择direct或adoption recovery；current settings B/C不得改写、推断或切换mode。Adoption R/p绝不能进入direct；pending-only link允许current whole-file raw/full identity与receipt original C witness发生任意non-v2 drift，只要求current strict JSON中的desired v2 subtree、static contract hash与commit token exact runtime-compatible，receipt original C witness继续作为immutable历史快照且不得更新。Adoption R+R/p同样只按runtime-compatible v2/static/token进入D。Direct R/p绝不能进入adoption，pending-only仍必须current raw bytes exact B且tuple-A/current-B non-v2重算相等，因为direct receipt仅durably授权并见证exact B，不能把其授权扩大成ambient content-level envelope；direct R+R/p只进入direct cleanup并可允许non-v2 drift。Mode/self-hash无法strict parse的R name是F incident，不根据settings形状猜mode。

专用`ambient_state_adoption`在读取authorization后先分类完整bounded inventory。在sole operation、canonical I final、无F/foreign/multiple/mismatch、R尚未final-only terminal的recoverable adoption范围内，以下A/B0/B/C/D互斥且完备；I/p、receipt temp、wrong mode/bytes/inode、final-only terminal和其它非法状态在该范围外，只读terminal或halt，绝不由adoption顺带收敛I：

- **A `first_adoption_receipt_publication`**：要求canonical V final nlink-1且V/p absent，R/R/p absent，preview-bound C双读稳定，adoption coordinate exact匹配preview与closed activation hash inputs并later/fresh/latest，且C/conflict、I/V/tuple、paths/D3/manifests、source closure、target/auth retained-fd/session bindings及control inventory完整有效。Write set仅为以第7节staged primitive创建exact `mode="ambient_state_adoption"` R temp、durably发布R/p、幂等link R final和unlink R/p；只有canonical R/p durable publication消费adoption coordinate。R/p durable publication后必须先cleanup own temp；cleanup完成后crash进入C，publication与temp cleanup之间crash留下R/p+same-inode temp nlink-2并由adoption recovery gate先移除redundant temp后进入C，R+R/p后crash进入D；不得写settings/I/V/audit。每个syscall前适用第6节完整auth/target+`source_guard(C_src)` prelude。
- **B0 `activation_absent_publication_before_adoption`**：要求canonical I final、V/V/p均absent、R/R/p absent、preview与phrase均exact绑定`activation_absent=true`、closed activation hash inputs及independently recomputed expected activation hash，current C在V写前single strict read exact匹配phrase且入口closure有效。Write set仅为从I tuple与committed static pins按确定性activation nonce公式渲染exact expected V，并以第7节staged primitive发布V/p、link V final、unlink V/p；不得接受caller nonce/hash/bytes，不得写settings、I或R。B0不消费adoption coordinate。V/p durable publication后必须先cleanup own temp；cleanup完成后crash进入B，publication与temp cleanup之间crash留下V/p+same-inode temp并由normal adoption convergence gate先cleanup后进入B，V+V/p后crash也进入B；V final后执行本invocation唯一一组back-to-back C双读与唯一一次bounded完整重验auth/source/session/I/V/R/inventory，并把该结果直接作为A前置，稳定才转A，漂移立即`NO_FURTHER_WRITE`、保留V并输出read-only conflict report。
- **B `activation_pending_convergence_before_adoption`**：要求canonical I final，R/R/p absent，且V为与phrase `expected_activation_object_hash` exact相等的pending-only V/p nlink-1或V+V/p same-inode/same-bytes nlink-2。Write set仅用第7节R4.2幂等link/unlink primitive把该expected V收敛为V final nlink-1；不得创建不同V、收敛I/p或创建R，也不消费adoption coordinate。完成后执行本invocation唯一一组back-to-back C双读与唯一一次bounded完整重验，并把该结果直接作为A前置；稳定才转A，漂移则`NO_FURTHER_WRITE`、保留V且不得创建R。后续invocation必须重新preview/auth；matching旧coordinate只有仍later/fresh/latest且逐syscall验证通过时可复用。
- **C `adoption_receipt_pending_convergence`**：mode-first分类必须得到R final absent、sole canonical adoption R/p nlink-1、canonical I/V final，receipt mode/self-hash/completion/post witness及operation/static/source/target/auth/D3/manifests/paths closure完整有效，且current strict settings仅须runtime-compatible exact desired v2 subtree/static contract/token；允许current non-v2/raw/full identity相对original C witness漂移，receipt内original C witness保持历史快照且绝不修改。Recovery coordinate使用第4节exact adoption recovery phrase并later than completion authorization、fresh/latest。Write set仅以`link_final_idempotent/v1`创建或确认same-inode/same-bytes R final并parent fsync/readback；不得unlink R/p或改变receipt bytes/mode/completion/post witness/settings/I/V。Link后本invocation无条件停止进一步mutation；这是控制流不变量，不新增任何receipt、gate、post dossier或其它durable `C_link_completed`字段。D只由后续invocation重新分类并执行其唯一一组bounded revalidation后进入；实现测试hook必须能断言同一invocation从未调用D的unlink路径。
- **D `adoption_receipt_final_pending_cleanup`**：mode-first分类必须得到canonical adoption R与R/p同inode/同bytes/nlink-2、canonical I/V final、receipt/self-hash及operation/static/source/target/auth/D3/manifests/paths closure有效；current settings只需strict runtime-terminal-compatible desired v2/static/token，允许non-v2 drift。Matching adoption recovery gate逐syscall仍later/fresh/latest时，唯一write set是`unlink_pending_idempotent/v1`、parent fsync和R final nlink-1 readback；不得改R bytes/mode/completion/post witness/settings/I/V。Current v2/static/token或closure漂移立即halt。

A receipt与post dossier只能陈述专用授权采纳锁内观察到的current exact v2/token state，必须写`cas_A_to_B_history_proven=false`、`actual_B_full_identity_recoverable=false`和reason `state_adoption_only_cas_A_to_B_history_not_proven`，不得声明CAS、C由B派生或actual B identity。并发adoption invocation各自受retained settings OFD及逐syscall transcript reverify约束；只有durably发布canonical R/p的coordinate成为completion authorization，loser看到stale coordinate、R或不属于自己的live temp后只能halt/只读terminal。

Direct receipt `mode="direct"`只允许normal CAS exact B readback或later mutating continue锁内确认current exact B、canonical I/V final且tuple-A/current-B non-v2重算相等后，以第7节staged primitive发布。它只绑定`actual_B_full_identity`和`actual_B_full_identity_recoverable=true`。Mode-first direct pending-only recovery要求R absent、canonical direct R/p nlink-1及current raw exact B，以matching `direct_receipt_pending_convergence` gate仅执行`link_final_idempotent/v1`；若current不是exact B，包括desired v2/static/token仍兼容但raw已成为C，R/p必须永久保持fail-closed pending并halt，runtime selected-zero，不能转adoption、改写receipt或靠content-level等价收敛。解除该状态只允许本文外部先恢复exact B，或未来另行设计、评审和授权的独立disposition protocol；本合同不授权该恢复或disposition。Mode-first `direct_receipt_final_pending_cleanup`要求direct R+R/p same-inode/same-bytes nlink-2、canonical I/V final、valid receipt closure及runtime-compatible v2/static/token，允许non-v2 drift，以同kind matching gate仅执行`unlink_pending_idempotent/v1`。两分支都不能改变receipt mode/bytes或completion kind；pending-only link invocation完成后停止mutation，cleanup只由后续invocation重新分类并执行其唯一一组bounded revalidation后进入。

每个invocation只执行一组back-to-back settings双读和一次bounded closure revalidation；A直接入口在R写前执行，B0/B在V mutation后执行并复用为A前置；任何persistent non-cooperative drift不触发自动loop、sleep/retry或重新render。尚未写入时返回`ZERO_WRITE_HALT`，已经发布V、R link或其它列明mutation后返回`NO_FURTHER_WRITE`，同时输出绑定冲突前后identity/hash、retained transcript状态和完整control inventory的read-only conflict report；operator要求人先停止/协调conflicting writer，再重新preview并取得fresh authorization。该外部并发liveness residual不保证eventual progress，但runtime在非terminal inventory保持selected-zero，既有final-only合法receipt保持只读验证，不存在不安全terminal。

Sole valid R final nlink-1且无pending/temp时，ordinary continue与adoption只能只读terminal verify；matching final+pending必须按persisted mode进入D或direct cleanup，runtime在收敛前selected-zero。Receipt后的current B或仅non-v2变化的C不创建第二receipt。任何settings v2/static/token或closure漂移、A+receipt、mode/witness错配或F均halt且不cleanup；Td只允许先走专用disposition，same-inode pending+temp只允许对应normal convergence gate，二者处置/cleanup后都需另fresh operation gate。

## 9. Receipt schema、completion authorization 与 post dossier

R4.2 receipt是 exact closed union，mode仅为 `direct`或 `ambient_state_adoption`。公共字段绑定 schema/revision、operation id、intent hash、activation hash、static contract、source、target/auth、D3/manifests、initial与 completion authorization、expected B、v2 hash、token、paths、runtime audit requirement、exactly-once和 self-hash；任何旧 `ambient_drift_recovery` mode、字段超集或 R4/R4.1混用都拒绝。

`completion_authorization`只记录canonical R/p由validated receipt temp经hardlink、parent fsync和strict readback首次durable publication时授权receipt bytes的coordinate，不记录temp create/write或后来负责link/unlink收敛的recovery gate。`direct` variant的completion kind仅可为`initial_execute`或`fresh_continue`：fresh execute在同一未中断initial全序中首次durably发布R/p可用initial coordinate；任何later crash recovery、exact-B receipt-only publication或其它mutating continue首次发布R/p都必须使用later fresh continue coordinate。`ambient_state_adoption` variant的completion kind只能为`ambient_state_adoption`并绑定A分支首次durably发布R/p的专用later fresh adoption coordinate；B0/B转入A时仍使用同一经再次验证的adoption coordinate。

Recovery gate是非持久、exact-residue-bound的幂等收敛能力而非receipt字段或invocation-local一次性票据。任何R/p或R+R/p先strict parse mode/self-hash，再由mode唯一选择`direct_receipt_pending_convergence`或`ambient_state_adoption_pending_convergence`；settings B/C没有mode选择权，direct completion中持久的`fresh_continue`也不是recovery gate。两种gate都必须later than receipt completion authorization，并在link、parent fsync、unlink及second parent fsync每个syscall紧邻前执行第6节retained transcript fresh/latest/target/auth+`source_guard(C_src)`完整prelude。Scope只授权同一residue可选的same-inode redundant temp unlink以及R4.2 `link_final_idempotent/v1`和`unlink_pending_idempotent/v1`；`EEXIST`/`ENOENT`只能按第7节重新分类成功。Gate可在同一或后续invocation重复用于exact convergence且不落durable consumption witness；过期或不再latest时可取得new later gate。所有分支不得改变completion kind/coordinate/hash、receipt mode、bytes或post witness；corrupt/unparseable R是F incident且不可recovery，跨op/receipt/path/inode/mode/action复用拒绝。Final-only terminal不要求gate；final+pending必须持有persisted mode对应gate，runtime在收敛前selected-zero。

`direct` witness只允许 `actual_B_full_identity`与 `actual_B_full_identity_recoverable=true`，禁止 C fields与任何统一历史 CAS声明。`ambient_state_adoption` witness只允许 `current_C_full_identity`、`current_C_raw_sha256`、`current_C_non_v2_jcs_hash`、`actual_B_full_identity_recoverable=false`、`cas_A_to_B_history_proven=false`和 fixed reason，禁止 `actual_B_full_identity`。两个 variant都不把未来 non-v2 settings identity变成 runtime authority。

Post dossier不提交为 execute输入，也不参与 operation id。它只能从已持久化且完整验证的 intent、activation与 receipt生成，记录 static contract、A identity/raw hash、expected B hash、token、receipt mode、direct exact B identity或 adopted current C identity/hash、source/session/auth closure、publication recovery history和 residuals。`publication_recovery_history`只能陈述从durable inode/nlink状态可验证的pending-only、final+pending与final-only收敛事实，不记录或推导recovery gate被消费、使用次数或invocation边界；协议没有这类durable witness。Adoption post dossier中的 C full identity/raw/non-v2 hash只是 receipt首次创建时 adoption witness的历史快照，不是当前 settings authority、持续不变断言或 runtime比较目标；receipt后 current C发生 non-v2 drift时不得更新 dossier或 receipt。Adoption post dossier只能声明 state adoption，不能声明 CAS发生或 B identity；缺 I/V/R时不得生成 `completed` post dossier，动态 CLI report不能替代 post dossier。

## 10. Runtime terminal contract 与权威 audit

Runtime对 enabled target session只承认完整 sole-operation closure：exact static-bound settings v2 subtree、tokenized r4Binding、intent、AUTHORIZED activation和 direct/adoption receipt全部 present、canonical、nlink-1、无 pending/foreign entry、hash/path/session/D3/source/static pins互相闭合。任一缺失、pending、forgery、schema mismatch或非法组合都返回 `selected_zero_injection`，且无 ADR0039 fallback。Sole valid final receipt与 current exact B或仅 non-v2不同的 C均按同一 terminal closure验证；receipt mode不因 current whole-settings raw identity变化而改变。

Runtime不要求 current whole-settings raw hash/full identity等于 expected B、direct receipt B identity或 adoption receipt C历史快照；它从 current strict duplicate-key-rejecting settings验证 desired v2 subtree、static contract hash与 token exact，并允许 receipt后的 non-v2变化，同时仍执行 target session exact selection、target/auth trusted JSONL prefix/coordinate、source/adapter/operator binding、old activation absence、halt/taint/pending rollback gate。Direct与 adoption receipt具有相同 runtime terminal完整性门，不因 adoption降低任何 closure要求；current v2/token任一变化立即 selected-zero。

Receipt完成只表示operator control graph terminal，不表示runtime injection已获授权。R4.2 runtime audit是receipt之后由runtime路径执行的独立`required_before_first_injection`权威门，durable object粒度为per-receipt；同一receipt永远只有一个expected immutable final object，不记录attempt、coordinate或injection history。每个准备执行first-injection decision的runtime invocation必须先生成第4节process-local `attempt_id`和non-authoritative `runtime_enable_preview`，再取得postdate该preview且逐字包含attempt id的new later/fresh/latest `runtime_enable_audit` coordinate并走完整durability gate；final object已存在只提供幂等内容，不提供当前enablement。Gate成功后该invocation只可作一次decision并process-local consume attempt；immutable object仍是per receipt而非per attempt，不能以旧process state、先前attempt或final object替代当前preview/gate，也不声称durable attempt witness。R4.2 idempotency key公式固定为：

```text
runtime_audit_idempotency_key = SHA256(
  UTF8("adr0040-d3-v2-session-start-r4.2-runtime-audit-idempotency/v1\0") ||
  ASCII(operation_id) || NUL ||
  ASCII(receipt_hash) || NUL ||
  ASCII(activation_object_hash) || NUL ||
  ASCII(static_contract_hash) || NUL ||
  ASCII(commit_token) || NUL ||
  UTF8(target_session_id) || NUL ||
  ASCII("allow_first_injection")
)
```

Runtime部署假设冻结为single-user：`runtime_uid == operator_uid == effective_uid`且runtime/operator effective gid相同；不满足即selected-zero，本文不声称隔离同uid恶意process。`audit_base` exact为preexisting `/home/worker/.pi/.pi-astack`，static contract必须冻结其`dev`,`uid`,`gid`,`mode`，runtime每次从anchored no-symlink ancestor chain打开并要求directory、fd/path same inode、metadata逐字等于pins，绝不创建、chmod、chown或修复base。R4.2 dedicated bootstrap是固定prefix automaton：`RB0=exact audit_base存在且dedicated parent adr0040-d3-v2-session-start-runtime-audit absent`→`RB1=parent以mode 0700、runtime uid/gid、base dev存在且empty，leaf r4.2 absent`→`RB2=parent entry集合逐字仅含leaf r4.2，leaf以mode 0700、runtime uid/gid、base dev存在且其inventory符合本节bounds/schema`。只有当前invocation fresh attempt-bound runtime gate下才可按RB0→RB1→RB2固定顺序执行anchored `mkdirat(0700)`并紧接对应parent `fsync`；每个mkdir和每个parent fsync前必须先完成attempt、coordinate、terminal closure、base与现有prefix metadata/inventory验证，再执行第6节紧邻auth/target+`source_guard(C_src)` prelude并立即进入syscall。Crash可留下mkdir成功但parent fsync未知的exact RB1/RB2 safe prefix；因为没有durable bootstrap-step marker，任何新invocation从preexisting RB1进入时都必须生成new attempt/preview/gate并先幂等`fsync(audit_base_fd)`/readback后才可创建leaf，从preexisting RB2进入时也必须先幂等`fsync(dedicated_parent_fd)`/readback后才可触碰root inventory或object，再推进唯一next state。错误顺序（包括leaf先于parent）、dedicated parent extra、wrong type/dev/uid/gid/mode、symlink、alias或`EEXIST` race mismatch一律selected-zero且不修复；RB1/RB2 directory创建与fsync不是audit object、coordinate consumption、attempt witness或injection marker。

R4.2 exact audit root是RB2 leaf `/home/worker/.pi/.pi-astack/adr0040-d3-v2-session-start-runtime-audit/r4.2`，final absolute path机械派生为`<root>/<idempotency_key>.json`，本invocation non-authority temp为`<root>/.<idempotency_key>.runtime-audit.stage.<invocation_nonce>.tmp`。Root必须retained anchored no-symlink、fd/path identity exact、runtime uid/gid、mode`0700`、dev exact audit-base pin且最多4096 entries；RB2既存时不得重新mkdir。Final必须bounded regular file、uid/gid exact runtime、mode`0600`、nlink-1，temp relation按本节与第4节。R4.2 audit object exact字段闭集为`schema_version`,`revision`,`idempotency_key`,`operation_id`,`receipt_hash`,`activation_object_hash`,`static_contract_hash`,`commit_token`,`target_session_id`,`injection_decision`,`audit_object_hash`；schema exact`adr0040-d3-v2-session-start-runtime-audit-object/r4.2-v1`，revision exact`R4.2`，decision exact`allow_first_injection`，`audit_object_hash = SHA256(RFC8785-JCS(object_without_audit_object_hash))`，persisted bytes exact JCS+single LF且<=1,048,576。没有`parent_hash`、append offset、line chain、cross-schema chain、partial row、multiple-row、coordinate或attempt字段；同key不同bytes/object/hash一律selected-zero。

Final absent时，runtime在independent cooperative serialization与retained anchored RB2 root fd下构造唯一expected bytes，按不可重排全序执行：branch入口完整HEAD/六artifact/closure revalidation与attempt-bound fresh gate/RB2 reverify→第6节prelude后`openat(O_RDWR|O_CREAT|O_EXCL|O_NOFOLLOW|O_CLOEXEC,0600)`创建unique temp并验证path/fd identity→每次完整write前完整prelude→`fdatasync(temp_fd)`前完整prelude→retained fd/path strict readback exact bytes/schema/self-hash/metadata/nlink-1→hardlink no-replace temp到final前完整prelude→`fsync(root_fd)`前完整prelude→strict final readback same inode/bytes、nlink-2→unlink exact temp前完整prelude→再次`fsync(root_fd)`前完整prelude→strict final-only readback nlink-1→以anchored `O_RDWR|O_NOFOLLOW|O_CLOEXEC` reopen final为`final_fd`并验证fd/path、expected bytes/schema/self-hash/metadata/nlink-1→`fdatasync(final_fd)`前完整prelude→最后`fsync(root_fd)`前完整prelude→strict final-only readback exact expected bytes/schema/self-hash/metadata/nlink-1。整个durability序列完成后仍不得直接进入injection，必须执行本节final-absent/final-existing共用的decision-time revalidation。Temp create/write/fdatasync/readback不是authority；final hardlink、parent fsync与readback构成durable per-receipt publication，但在temp cleanup与第二parent fsync/readback完成前仍不得注入，且从不构成per-attempt durable witness。

Hardlink返回`EEXIST`必须重新分类。仅当final与current exact temp same inode/bytes/nlink-2，或final为separate inode但bytes exact等于唯一expected object、metadata/nlink-1正确且current temp仍是exact expected nlink-1时可幂等收敛；前者unlink redundant link，后者unlink current temp，均在每个syscall前执行第6节完整prelude、fsync root并strict readback final-only。EEXIST收敛本身仍不足以注入，随后必须在same current fresh gate下对final执行带完整prelude的fdatasync→root fsync→strict readback，再进入共用decision-time revalidation。Final同key但不同bytes、self-hash、binding或schema即selected-zero，不覆盖、不unlink final。Crash遗留temp若bytes/key/schema/self-hash/metadata exact expected且final absent或为上述exact EEXIST关系，可在next runtime invocation生成new attempt/preview并取得post-preview fresh `runtime_enable_audit` gate后继续收敛，但必须先anchored reopen并在current gate下重新`fdatasync(temp_fd)`及strict readback后才可hardlink；partial、wrong或unreadable temp没有partial chain语义，只能使用第4节专用`runtime_audit_temp_disposition` phrase按path/key/dev/ino/raw descriptor unlink+fsync后停止，再由另一new fresh runtime gate重试。不得触碰其它key、object或inode。

Final已经可见时也必须由当前invocation生成new attempt与preview并取得post-preview later/fresh/latest exact gate，在branch入口先完整重验HEAD、六artifact与全部closure，再以`O_RDWR|O_NOFOLLOW|O_CLOEXEC`打开并同时验证fd/path、exact expected bytes/hash、schema/self-hash、uid/gid/mode/nlink-1；随后在current attempt-bound gate下依序于`fdatasync(final_fd)`与`fsync(root_fd)`各自紧邻前执行第6节完整prelude，再作strict final-only readback。该fresh durability refresh全序完成后仍不得直接进入injection，必须执行下一段final-absent/final-existing共用的decision-time revalidation；“可见”等于“durable”或“此前已audit”都不是充分条件。任何auth append/expiry、attempt mismatch/reuse、target/auth/source/terminal drift、bootstrap顺序或metadata错误、EEXIST mismatch、oversize、owner/mode/link/symlink或inventory异常均selected-zero、bounded退出且prompt/canvas保持未注入；作出selected-zero decision后同一attempt也不能在本invocation重试第二次decision。

无论final absent materialization、EEXIST convergence还是final existing refresh，在最后一次`fdatasync(final_fd)`、最后一次`fsync(root_fd)`及其strict final-only readback全部成功之后，且在prompt、canvas或injection发生任何可观察副作用之前，runtime必须紧邻执行一次不可跳过、不可复用旧结果的decision-time revalidation。该检查机械重读并证明：retained authorization attempt coordinate的exact text/hash仍逐字匹配当前`runtime_enable_preview`，仍postdate preview boundary、later than receipt completion、fresh/latest，retained `attempt_id` exact且尚未process-local consumed；current strict duplicate-key-rejecting settings的desired v2 subtree、`static_contract_hash`与`commit_token` exact；sole canonical I/V/R均为final-only nlink-1且bytes/schema/self-hash/operation/source/static/target/auth/D3/manifests/paths逐项闭合，无pending/temp/foreign，old activation absent，halt/taint/pending rollback及rollback-root terminal gates exact；audit final仍为唯一expected final path/bytes、exact schema/self-hash/uid/gid/mode且nlink-1；最后执行完整`source_guard(C_src)`并要求`HEAD^{commit} == C_src`、六artifact及全部closure live hashes exact。整个decision-time revalidation与允许的首次prompt/canvas/injection副作用之间不得有日志、inventory、settings/session读取、状态转换、重试或其它协议步骤。任一项失败都作出`selected_zero_injection`，prompt/canvas保持逐字未变，并在返回前把该attempt process-local标记consumed以禁止同一invocation重试；全部通过时只允许紧接的一次first-injection decision，decision edge将attempt process-local标记consumed并拒绝第二次decision。该门对final-absent与final-existing路径完全相同。

Legacy R4.1 audit file exact `/home/worker/.pi/.pi-astack/adr0040-d3-v2-session-start-runtime-audit.jsonl`仅是immutable read-only history；R4.2 terminal gate不append、不chain、不迁移、不要求其存在或有效，也不从其中任何row推导R4.2 authorization。诊断工具若读取它只能按frozen R4.1 schema独立报告，结果不能allow或deny R4.2 injection。`operator-audit.jsonl`仍是operator在authority closure完成后的可选best-effort非权威日志，与R4.2 immutable runtime audit root在path、owner、schema、durability和授权语义上完全不同；任一对象不得替代另一方。Terminal read-only continue只返回`terminal_verified`与`runtime_audit_required_before_first_injection=true`，不得创建/刷新runtime audit、追加authority或消费coordinate。Pending intent/activation/receipt包括final+pending pair始终selected-zero；runtime不cleanup control、不补receipt、不执行adoption。Runtime enablement必须由未来实际runtime调用中的独立fresh coordinate提供，本design-only文档不构成该授权。

## 11. Preview 与执行输入边界

Committed preview是static template，只展示top-level static contract hash、固定paths/pins、allowed v2 prestate、desired v2 template内唯一static/dynamic sentinels、transformer/mutation/conflict contract、source closure policy、closed `source_commit_binding="dynamic_preview_and_initial_phrase"`、预计动态字段名、zero-write assertions和`S2_NOT_AUTHORIZED`。其`dynamic_field_names`必须包含future dynamic field name `source_commit`，但template不提供该字段的值或任何具体HEAD OID；它不捕获live A、不预测live B、不记录dynamic coordinate/token/operation id，不因unrelated settings edits或HEAD推进而需要重写。

CLI default mode生成动态、非权威、零写report，可显示当前是否满足P(A)、当前v2 prestate、non-v2 JCS hash、source/session/auth/control readiness和“若获得未来合法coordinate将动态捕获”的说明。专用initial preview mode按第4节额外验证同一现场commit上的六artifact与content-only closure，捕获dynamic `C_src`并输出`initial_dynamic_preview`及exact phrase；该preview仍不得输出可执行authorization tuple、不得持久化intent-ready capsule、不得声称B/token/operation id已授权。即使为了可读性模拟transform，也必须标为unbound simulation且不产生authority hash。

`runtime_enable_preview`是runtime invocation在terminal closure read-only验证后生成的另一种non-authoritative zero-write report，不是committed preview、execute input或authorization。它必须由当前invocation现场生成attempt并保留process-local attempt/preview transcript boundary后输出；caller-supplied preview、attempt、preview hash或phrase不能作为输入。Runtime只能接受postdate该preview boundary且逐字匹配其actual phrase的coordinate，preview自身不允许audit bootstrap、temp/final mutation或injection decision。

Execute/continue绝不接受preview/report文件、stdin report、caller-supplied A/B/token/tuple/operation id、环境变量override或“已审阅”flag作为输入。Execute只从committed content-only static contract/manifests、live trusted sources与auth transcript中的exact initial phrase机械重建phrase所绑定的preview transcript prefix boundary和dynamic `C_src`，并在锁内自行捕获A；它不信任caller提供的preview bytes。Continue只读取committed static closure、persisted intent tuple/activation/receipt candidate与fresh trusted continue coordinate，并要求phrase和live `HEAD^{commit}`都逐字等于persisted `C_src`，且任一mutating branch入口及每个mutating syscall前分别完成full source revalidation与`source_guard(C_src)`。

Preview zero-write验收同时覆盖settings、完整target session、D3、control、rollback、old activation、runtime/operator audit、quarantine、Git metadata；initial dynamic preview还必须覆盖六artifact与全部closure rows在同一captured `C_src`上的tracked/nonignored/live==`C_src`/clean验证。Active authorization transcript继续采用R4.1 frozen-prefix append-tolerant attestation并从generic exact snapshot排除。默认preview、initial dynamic preview及其dynamic report在strace下不得出现mutating filesystem syscall。
