## 1. Executive summary

P0.5 initial implementation (commit `fec969e`, 2026-05-23) shipped
multi-view verification with 7 transient fallback paths that silently
fell back to proposer direct-write — violating ADR 0025 §3.1 A' hard
constraint ("non-trivial create / destructive ops MUST be
double-reviewed"). 3-family T0 review (Opus 4-7 / GPT-5.5 / DeepSeek
v4-pro) identified 4 P0 blockers + 7 P1 issues + DEFER design hole.

Batch 1-3 (16 commits across `b575eab` → `1a6f6f7`) closed the A' gap
+ implemented unified staging-pending-multiview fallback queue.

**A' layer is now closed**: every transient failure path (reviewer
unavailable, pass1/2 call failed, pass1/2 unparseable, pass2 deferred)
either stages the candidate for replay OR drops it after audit. Six
silent-fallback paths are converted to staging; the seventh
(`confirm_pass1_not_synthesizable`) keeps deliberate skip because it
represents a known Pass 1 schema limitation (P1.5 plan to upgrade).

## 3. A' layer closure proof (post-batch 3)

### 3.1 Decision matrix — what writes to brain after batch 3?

| `runMultiView` exit | `final_decision` | Brain write executed? |
|---|---|---|
| `triggered=false` | `proposerDecision` | YES — but A' doesn't cover (trigger criteria said "low-value", reviewer not required) |
| `triggered=false` + replay path (batch 3c-i.5 R1) | `op=skip(multiview_trigger_disappeared_on_replay)` | NO — replay drops candidate |
| `reviewer_unavailable` | `op=skip(multiview_staged_for_replay)` | NO |
| `pass1_call_failed` | `op=skip(multiview_staged_for_replay)` | NO |
| `pass1_unparseable` | `op=skip(multiview_staged_for_replay)` | NO |
| `pass2_call_failed` | `op=skip(multiview_staged_for_replay)` | NO |
| `pass2_unparseable` | `op=skip(multiview_staged_for_replay)` | NO |
| `verdict=confirm_proposer` | `proposerDecision` | YES — reviewer agreed |
| `verdict=confirm_pass1` + synthesizable | synthesized | YES — reviewer decided |
| `verdict=confirm_pass1` + workflow-lane refusal | `op=skip(multiview_workflow_lane_protected)` | NO |
| `verdict=confirm_pass1` + not synthesizable | `op=skip(multiview_pass1_op_not_synthesizable)` | NO |
| `verdict=defer` | `op=skip(multiview_staged_for_replay)` (batch 3b promoted) | NO |
| Signal abort mid-flight | `op=skip(staged)`, then replay deletes new staging (3c-i.5 R4) | NO |

### 3.2 Each `YES`-row above is A'-compliant

- `triggered=false`: A' doesn't apply (decision wasn't high-value to begin with). Verified by `shouldTriggerMultiView` (multi-view.ts:105-224) carrying the trigger gate logic.
- `confirm_proposer`: Pass 1 reviewer disagreed with proposer; Pass 2 reviewer (reveal) sided with proposer after seeing both. Two independent API calls (different family from curator) confirmed → A' definition met.
- `confirm_pass1` synthesizable: Pass 1 + Pass 2 both ran; reviewer's verdict (not proposer's) is what gets written. A' met by construction.

### 3.3 Each `NO`-row is staging-tracked OR audited

- Six `staged_for_replay` rows: candidate written to `~/.abrain/.state/sediment/staging/multiview-pending-{hash8}-{iso}.json`, replay attempts on next `agent_end`.
- `workflow_lane_protected` / `pass1_op_not_synthesizable`: audit row records the skip with reason; candidate is dropped (not staged) because the issue is a writer-side hard constraint or known schema limitation that won't be resolved by retry.
- `trigger_disappeared_on_replay`: replay's defensive A' guard (batch 3c-i.5 R1) — if `shouldTriggerMultiView` returns false at replay time (e.g. neighbor archived, candidate confidence no longer ≥ 8), the candidate is dropped rather than direct-written. Audited with explicit `multiview_trigger_disappeared_on_replay` reason.

## 6. v1 limitations (deliberate, transparent)

### 6.1 `writeApprovedToBrain` stub

Replay's `writeApprovedToBrain` callback (sediment/index.ts Lane R)
audits `multi_view_replay_would_write` + `candidate_lost: true` but
does NOT actually dispatch to the writer. Real implementation requires
extracting the 300-line op dispatcher from `sediment/index.ts:2280-2440`
into a shared function — significant refactor risk that is deferred
to a follow-up batch.

Consequence: replay can decide `op=create/update/merge/...`, write a
`would_write` audit row, then drop the candidate. A' constraint
still holds (reviewer agreed, but persistence didn't happen). Dogfood
users will see clear audit trace with `candidate_lost: true` flag for
forensics.

### 6.2 kill switch reused for replay

`if (classifierEnabled && cwd && sessionId)` (sediment/index.ts Lane
R) — `classifierEnabled = settings.autoLlmWriteEnabled !== false`.
This is the same kill switch as the correction classifier. Users who
disable autoLlmWriteEnabled get no replay either. v2 may add a
separate `multiViewReplayEnabled` setting for orthogonal control.

### 6.3 `confirm_pass1_not_synthesizable` dead-loop potential

This path keeps `op=skip` (not staging). If the classifier produces
the same candidate again on a future turn, multi-view will trigger
again, Pass 1 will recommend the same un-synthesizable op, skip
again. Cost: ~$0.005-0.02 per redundant cycle. Mitigation deferred
to P3 (candidate-identity throttle: SHA256(compiledTruth) cache last
N hours of "not synthesizable" decisions; classifier skips if hit).

### 6.4 Tier 1 smoke not written

ADR 0025 §6.1 Tier 1 lists "multi-view 触发条件命中时确实跑了两次独立 API 调用 (log 验证)" — the specific automated assertion most relevant to batch 1-3 work. **Not implemented in batch 5**. Reason: writing it requires either:
- (a) a mock reviewer model registry that returns deterministic JSON outputs (similar to mockModelRegistry pattern in `smoke-memory-sediment.mjs:597-642`), driving through full `runMultiView` to assert 2 separate `streamSimple` calls, OR
- (b) a fixture branch + recorded reviewer outputs, replaying through `runMultiView` with stubbed `callReviewerModel`.

Either is ~150-250 lines of test scaffolding. Deferred to a follow-up
phase. The 7 single-batch reviewers (across 5 model families) provide
ad-hoc verification in lieu of automated smoke; this dossier records
that verification trail.
