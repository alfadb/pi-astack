## 0. 元数据：为什么这份 audit 比 ADR 0027 本身更重要

这次研讨的真正产出**不是 v0.2 文本本身**，而是验证了以下方法论可重复：

1. **跨 provider 多轮辩论可以系统性消除 v0.1 framing 偏置**——R1 三家一致打回"正交"措辞（C1 误用）、一致打回"全 prompt-native"（C3 误用）、一致打回"拒绝 fire-and-forget"（C4 措辞过粗）。这三处偏置主会话 LLM 单独 review 时没有发现。
2. **趋同辩论是有限轮次可终止的**——R1 → R2 → R3 的让步轨迹清晰，每轮都能识别"哪些是真分歧、哪些是表述分歧"。R3 后剩余分歧只有 framing 中一个词的取舍（Stigmergic 进不进主名），三家 ACCEPT 即可结束。**不需要 R4**。
3. **v0.2 真实价值是"诚实标边界"而不是"新范式"**——R1 三家一致指出 v0.1 列的"hub-driven dynamic role / stigmergy / 自然语言 plan"业界都在做；真独特性是 §5.1 表上的四条（L1/L2 不变量分层 / 跨 provider 防 RLHF 共谋 first-class / AI-Native 三态显式方法论 / completion semantics 显式补丁）。**理论原创性评分从 7.3 降到 7.0 是正确的——稀释的部分本来就不该 claim**。

跟 ADR 0024 audit 的对照：
- ADR 0024 audit 揭示了 **leading-prompt 偏置 + RLHF 机械主义偏置** 两层偏置（R1-R6 6 轮才校正）
- ADR 0027 audit 验证了 **R1 注入 §3 AI-Native bias self-check 后 RLHF 偏置不再触发**（三轮即收敛）

这进一步实证了 ADR 0024 R7 的发现："硬注入 AI-Native 自检后，三家评审产出 21/21 PE-form 建议，无一落入机械门"。本次 ADR 0027 评审延续了同款防御机制——三轮 prompt 都含 Step 0 bias self-check，三轮产出**无一条**推荐"加 schema 阈值 / 加机械关卡作为 LLM 行为层防出错主路径"。

### 2.3 DeepSeek v4-pro R1 关键产出

**Step 1 立场**：C1 HOLD / C2 MODIFY / C3 MODIFY / C4 HOLD

**最大独家贡献**：stigmergy 结构性盲区识别（H1 起源）
> stigmergy 的固有特征是**异步、无确认**——worker A 把结果写入 trace，hub 不知道 worker A 是 done 了还是 crash 了还是还在 slow。这是 stigmergy 作为协调模式的结构性盲区，不能靠 "在关键边界写 trace" 解决——因为 crash 了的 worker 根本不会写 "我 crash 了" 的 trace。

**第二大独家贡献**：hub dynamic role assignment 零样本 learning（H5 起源）
> hub 面对一个新任务，没有任何可参考的历史分配记录，必须 "即兴决定派谁"。hub 的即兴决策没有 baseline 可对照——它怎么知道应该派 3 个 worker 还是 8 个？怎么知道 worker 应该专注于代码探索还是方案设计？

**第三大独家贡献**：multi-role identity drift（P1 起源）
> ADR 0024 设计假设单用户单身份。但用户可能有多个角色——在家是 OSS contributor（偏好宽松的代码风格），在公司是 team lead（偏好严格的 review 流程）。当前设计把 "用户" 当成单一身份。如果一个 preference（"代码风格宽松"）来自 OSS 项目，但 L2 在公司项目的决策简报里引用了它——这就是 identity drift。

**Step 7 评分**：原创性 8 / 一致性 9 / 落地 5 / 发表 7

### 3.2 R2 反驳交锋

**Opus 反驳 GPT-5.5（B2 — sympoietic 保留）**：
> sympoiesis 学术定义是 *non-self-producing, no spatial/temporal boundary, collectively-producing systems*。pi-astack L1 sediment 显式自维持（curator+aggregator+writer 构成闭环写 lane，ADR 0025 §1.1），有明确边界（七区 + git repo）。这是 autopoiesis 不是 sympoiesis。若发表 / 外评，STS 学界会以"术语挪用"开炮，**论据弱点是真的，不是嘴硬**。

**DeepSeek 反驳 Opus（B1 — swarm 词放弃）**：
> swarm 不必严格等于 large-N homogeneous insects；在 agent engineering 里，small-N heterogeneous agent swarm 也是常见用法。但 Opus 的提醒有效：如果标题叫 "Swarm"，会把读者注意力错误引向大规模同质群体。**结论：top-level 不用 swarm；机制层（C2'）继续用 stigmergy。**

**GPT-5.5 反驳 DeepSeek（B2 — L1 only stigmergy 太纯化）**：
> "L1 只走 stigmergy" 太纯化。ADR 0025 里 L1/sediment 现实包含 `agent_end` hook、Lane C fire-and-forget、recursive drain loop、locks、git commit、audit append、staging loader、scheduler/aggregator。这些不是全都能干净归入 stigmergy。更准确说法是：**L1 的 authoritative coordination state 应沉积在 stigmergic substrate；但运行机制可以包含 hooks、locks、schedulers、singleFlight、background jobs**。

## 4. R3 — 终局收敛 + framing 投票 + 共识签字

每家看 R2 全文后做最终表态。

### 4.1 三项遗留分歧最终表态

**H1（stigmergy completion semantics 盲区）**：

- **Opus ACCEPT**：「stigmergy 的语义底座是『共享状态被读到』，但『读不到新状态』在物理上无法区分三种情况：(a) 生产者还没产出、(b) 生产者已死、(c) 生产者卡在 LLM 推理里。这不是实现 bug 而是 paradigm 缺口——artifact mutation 自身无法编码 liveness。」
- **GPT-5.5 ACCEPT**：「stigmergy 对『任务是否真正完成 / 谁负责收口 / 失败是否已传播』存在 completion semantics 盲区；L2 rendezvous 场景需要 message passing / heartbeat / ack 作为补充。」
- **DeepSeek**（原提出方）：「Opus/GPT 若在 C2' 中明文纳入 `completion semantics 盲区 + heartbeat 必需` 作为限定条件，我的核心关切已被结构性吸收。盲区承认本身就是治理补丁。」

**H2（swarm 词 top-level 去留）**：

- **Opus 让步**（部分）：「复盘 DeepSeek 引的 MAS 文献：广义 MAS 不预设 large-N，但 `swarm` 这个具体词在 Bonabeau/Dorigo 谱系里确实预设 large-N + homogeneous + emergent。当代 agent 语境（LangGraph / AutoGen / Claude swarm）`swarm` 词义已显著漂移、不再 strictly large-N。**结论**：保留 `swarm` 在 top-level 名字里有真实歧义代价、又没有显著表达增益。让步——top-level 不用 swarm。」
- **GPT-5.5**：不让步。「我仍认为 swarm 必然预设 large-N 同质；但它确实容易被读成 large-N、弱中心、同质 worker。因此我不建议把 Swarm 放进 top-level 名字，可保留为 L2 子系统名。」
- **DeepSeek 让步**：「Opus 的反对有合理语言学依据——`swarm` 在工业界已被 Kubernetes schedulers / Ray 绑定到同质大规模语义，继续坚持会制造不必要的误解税。为共识让渡。」

**H5（hub baseline 必须先有 evaluation harness）**：

- **Opus ACCEPT**：「Hub dynamic role assignment 是 L2 的 single point of failure：assignment 错 → 所有下游 sub-agent 在错误前提下 stigmergy，错误被放大而不是被纠正。这一点比 H1 更严重，因为 stigmergy 默认 + heartbeat 兜底解的是 liveness，hub 错配解的是 correctness。**没有 evaluation harness 就上线 hub baseline 等于把 ADR 0024 §3 的『AI-Native 不等于不验证』反过来读**。Hub baseline blocking 成立。」
- **GPT-5.5 ACCEPT-限定**：「hub dynamic role assignment 若进入 baseline，就必须有 evaluation / rollback / degradation 语义后才能上线；否则只能作为实验性能力，不进 v0.2 baseline。」
- **DeepSeek ACCEPT-条件满足**：「若 v0.2 明文写入『Hub dynamic role assignment baseline ADR 未完成前，L2 swarm 不进生产』，这比我的原始 blocking 立场**更强**——它不仅承认阻塞，还给出了 gate 条件。」

### 4.2 Framing 终局投票

| 投票 | 投票方 | 理由 |
|---|---|---|
| **CDLAS** Coupled Dual-Loop Agent System（无 Stigmergic） | Opus | "Coupled Dual-Loop" 精确描述 C1'（两套 invariant + 互嵌共生）+ C6（共享锚点）的双重含义，stigmergy 落到机制层正好。STTLS 把 stigmergic 顶上来过度承诺一种机制。 |
| **CSDLAS** Coupled Stigmergic Dual-Loop Agent System | GPT-5.5 | 保留 dual-loop 治理边界，加入 stigmergic 回应 DeepSeek，避免 Swarm 作为主名带来的 large-N 误读。 |
| **CSDLAS** Coupled Stigmergic Dual-Loop Agent System | DeepSeek | "Coupled" 精确捕获 L1/L2 互嵌（我在 R1/R2 的核心坚持）；"Dual-Loop" 区分于单体 agent 的 inner/outer loop；"Stigmergic" 保留；"Agent System" 不加权预设规模。四个词各司其职，无一冗余。 |

**结果**：2/3 多数 → **Coupled Stigmergic Dual-Loop Agent System (CSDLAS)**

Opus 在 ACCEPT-with-reservations 中接受 CSDLAS 作 baseline，保留意见已记入 ADR 0027 §11.4。

| 理论原创性 | 7.3 | 7.0 | -0.3 | swarm/stigmergic 退出 top-level 后稀释；但真独特性（§5.1 四条）保留更精确 |
| ADR 0024 一致性 | 8.0 | 9.0 | +1.0 | C3' 把认知/infra 分层显式开，根治 v0.1 把 §3 套到 L2 infra 的过度套用 |
| 工程可落地 | 5.0 | 8.0 | +3.0 | C5（fail/degrade/cancel/resume）+ C6（causal trace anchor）+ 三条子 ADR 拆分 + H5 blocking gate |
| 值得发表 | 6.3 | 8.0 | +1.7 | "completion semantics 是 stigmergy 结构盲区 + dual-loop coupled 而非 orthogonal" 是可单独成文的小贡献 |

**核心提升**：从"我们提出一个新范式"变成"我们诚实标出范式的边界、缺口和必须的补丁"——这反而更符合 ADR 0024 自身的方法论气质（"先承认机械工程兜底合法，再要求 LLM 行为层走 prompt-native"）。

## 6. 方法论对照：ADR 0024 audit vs ADR 0027 audit

| 维度 | ADR 0024 audit（R1-R6） | ADR 0027 audit（R1-R3） |
|---|---|---|
| 起始偏置 | leading-prompt 偏置 + RLHF 机械主义偏置（两层） | 无（R1 起即注入 §3 AI-Native bias self-check） |
| 校正轮次 | R1-R6 6 轮（R4 balanced framing 校正 leading-prompt；R6 元层 reframe 校正 RLHF 机械主义） | R1-R3 3 轮（趋同辩论） |
| 评分轨迹 | 27.7% → 22.3%（R3 偏置低点）→ 39.3%（R4 critic）→ 54.3%（R4 steelman）→ 43%（R5 + UU）→ 55-60%（R6 reframe） | 5.0（R1 起点）→ 8.0（R3 v0.2 落地评分）单调上升 |
| 关键发现 | 揭示 multi-LLM review pipeline 两层偏置 | 验证"R7 Layer F 自检"防御机制可重复（R1 三轮 prompt 都含 Step 0，无一条机械门方案产出） |

**两份 audit 一起构成 pi-astack multi-LLM review 方法论的双锚点**：
- ADR 0024 audit 是"什么偏置会发生 + 怎么发现"
- ADR 0027 audit 是"防御机制 ship 后是否真有效"

R7 时的 21/21 PE-form 实证在 ADR 0027 R1-R3 三轮**继续保持 0/N 机械门方案**——这是同款 maxim 的第二次实证。建议未来每次 multi-LLM review 都延续此格式，作为 RLHF 偏置长期监测样本。

## 7. 给未来 multi-LLM review pipeline 的几条建议

基于本次 R1-R3 经验：

1. **R1 prompt 必须包含 Step 0 bias self-check**——明确要求 reviewer 复述 ADR 0024 §3 核心一句话 + 承诺不默认推荐机械门
2. **R1 立场分布 4 全同（4 HOLD / 4 REJECT / 4 MODIFY）是危险信号**——必须 Step 6 自查（本次 GPT-5.5 自查后仍坚持 4 MODIFY 但给出了具体论据，可接受）
3. **R2 应明确要求"反驳 + 让步 + 趋同提案"三段，不只是再写一遍 R1**——本次 R2 让步轨迹清晰是因为 prompt 显式问"哪些点你愿意让步"
4. **R3 应该是 framing 投票 + v0.2 完整文本锁定 + 共识签字**——明确终止条件，不让讨论无限蔓延
5. **不超过 5 轮**——本次 R3 后剩余分歧只是 framing 中一个词的取舍，三家 ACCEPT 即可结束。R4/R5 边际效用必然递减
6. **跨 provider 强制**——三家都来自不同 provider（Anthropic / OpenAI / DeepSeek）才能防 RLHF 共谋
7. **总成本应 < $5**——本次 ~$2.07 / ~12 分钟 / 三轮并行 dispatch 是合理预算上限
