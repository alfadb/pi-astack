# Cross-Reviewer Consensus — P0 (BLOCKING)

## P0-α : Sub-agent `toolResult` contaminates main-session sediment input

**Severity**: maximum. INV-IMPLICIT-GROUND-TRUTH silently violated for `dispatch_parallel` workloads.

### Evidence

- **GPT-5.5** (strong): "PR-B 只阻止 sub-agent 自己的 agent_end，没阻止父会话 sediment 读 sub-agent toolResult"
- Code:
  - `sediment/checkpoint.ts:269-270` `entryToText()` renders `role === "toolResult"` into sediment window
  - `sediment/checkpoint.ts:289+` `buildRunWindow()` consumes branch without filtering toolResult
  - `sediment/index.ts:961-965` comments themselves acknowledge: subagents return output via tool_result; real session's own agent_end will see subagent content, including MEMORY: blocks
- **DeepSeek** (death-knot 2): `dispatch_parallel` 16 tasks → tool_result text → main-session conversation window → Lane C extractor reads → sub-agent reasoning "Based on the user's preference for pnpm..." is captured as user-implicit-truth signal in brain. **16× noise per fan-out**.
- **DeepSeek** (death-knot 3, escalation): sub-agent has `memory_decide` → reads brain → produces interpretation → interpretation returns via tool_result → sediment extracts → writes back to brain. **Self-exciting loop**. ADR 0026 §3.4 echo-chamber breaker (检测同一 entry 连续 5 次 DECISIVE) does NOT detect this — the breaker watches *usage* of entries, not *creation* of phantom entries by sub-agent interpretation.

### Why this matters

PR-B's `isSubAgentSession(ctx)` correctly blocks sediment's `agent_end` handler when fired INSIDE the sub-agent's own session. But the sub-agent's output flows back to the parent session as `tool_result` messages, which are stored in the parent's branch. When the parent session's own `agent_end` fires, sediment's Lane C reads the parent branch — which now contains the sub-agent's reasoning text — and extracts memory candidates from it.

The result: **statements generated by an LLM sub-agent are learned as if they were user implicit truth signals**.

### Suggested fix direction

Three options surfaced by reviewers:

1. `buildRunWindow` excludes `toolResult` blocks originating from `dispatch_agent` / `dispatch_parallel` calls (cleanest; needs branch entry to carry `tool_name` provenance)
2. Extractor prompt explicitly tags "the following blocks are sub-agent reasoning, NOT user statements; do NOT extract them as user preferences" (defense at prompt boundary; LLM compliance variable)
3. Introduce a separate `l2_outcome` schema for sub-agent results and let only a designated `l2-outcome-summarizer` consume them (clean separation; larger refactor)

## P0-β : Anchor captured at write-time = wrong turn for async writers

**Severity**: maximum. C6 in long-running async writer scenarios is **broken**, not just "best-effort".

### Evidence

- **DeepSeek** (death-knot 1): Lane C extractor is fire-and-forget (~60s LLM call). User submits next prompt → `before_agent_start` fires → `_currentTurnId` bumps N → N+1 → extractor completes → `getCurrentAnchor()` returns turn N+1's anchor. **The audit row for sediment work triggered by turn N carries turn N+1's anchor**. C6's claim "(session_id, turn_id) is the only required join key" is broken by this.
- **GPT-5.5**: "C6b 目前是 observability retrofit，不是严格 causal provenance"
- **Opus** (Step 5-1): identified but only listed as risk, not P0
- The C6b commit message itself acknowledges this trade-off — three reviewers independently agreed acknowledgment is not enough.

### Suggested fix direction

Snapshot anchor at trigger time (synchronously inside `agent_end`), pass it through to the async writer rather than letting the writer call `getCurrentAnchor()` itself.

```ts
pi.on("agent_end", async (event, ctx) => {
  const triggerAnchor = getCurrentAnchor();  // snapshot HERE, before any await
  scheduleBackground(triggerAnchor, ...);
});

// background worker:
async function backgroundWork(triggerAnchor, ...) {
  // ... do LLM call ...
  await appendAudit(root, event, { anchorOverride: triggerAnchor });
}
```

`spreadAnchor(anchorOverride ?? getCurrentAnchor())` keeps backward compatibility. For deferred fire-and-forget (`abrain/git-sync.ts` `pushAsync`), write BOTH `trigger_anchor` (snapshotted) and `write_anchor` (live) so consumers can distinguish.

All three reviewers concur. Reviewer A's Step 4 contains the most concrete C5 v1 design sketch (heartbeat protocol + `terminal_state` schema + cancel protocol stub). Not repeated here; see Reviewer A Step 4.3 in §A below.

## Step 4 — C5 missing impact

### 4.1 Can ship L2 production now?

**Cannot ship L2 mutating main path; can ship L2 read-only experimental/audit path** — exactly what ADR 0027 §8 cost #2 writes as "red line".

**Current de facto safety net**:
- `MUTATING_TOOLS = {bash, edit, write}` default disabled, requires `PI_MULTI_AGENT_ALLOW_MUTATING=1`
- `DEFAULT_TIMEOUT_MS = 30min` — timeout is sole liveness backstop
- nested dispatch prohibited → no fan-out explosion

**This is lucky read-only, not C5**. Any PR flipping the three defaults ("let worker also edit this staging directory"), L2 immediately enters ADR 0027 §8 cost #2 red zone.

### 4.2 Scenarios exposing C5 absence (concrete dogfood cases)

1. **Multi-model audit stuck scenario**: dispatch_parallel launches 3 cross-provider reviewers running 30min audit, one provider rate-limit hangs but pi-ai retry loop doesn't trigger `auto_retry_end` → main session waits full 30min timeout. User sees dispatch footer running with no heartbeat distinguishing "sub-agent alive" vs "sub-agent hung". → **missing heartbeat (C2') + missing cancel (C5)**.

2. **L1 triggers L2 sub-agent then user Ctrl-C main session**: current dispatch handles abort signal via session.abort(), but **doesn't write `terminal_state: cancelled` row to audit** — only `result: "fail" + failure_type: "aborted"`, lacks "real failure vs user cancel" distinction for outcome attribution. → **missing cancel explicit schema (C5)**.

3. **multi-view P0.5 staging replay failure retry**: ADR 0025 §4.4.6 mentions replay decision op!=skip causes candidate loss. This is missing resume — if sub-agent reviewer halfway through staging replay pi restarts, no idempotency_key for next session to resume from checkpoint. → **missing resume (C5)**.

4. **Degraded return scenario**: future some sub-agent task produces "partial result + I didn't finish reading whole codebase", current audit only records success/fail binary, can't record `terminal_state: degraded + what_dropped`. LLM gets "partial output" mistakes for complete result. → **missing degrade (C5)**.

### 4.3 C5 design sketch

**Heartbeat protocol (C2' + C5 coexist)**:
```
extensions/_shared/heartbeat.ts:
  - Each dispatch spawn worker creates heartbeat trace channel
    `${projectRoot}/.pi-astack/dispatch/heartbeat/${session_id}_${turn_id}_${subturn}.jsonl`
  - Worker every N seconds (default 15s) via subscribe() sees message_start /
    tool_call_start events, appends a line `{ts, type: "alive", phase}`
  - Caller polls last line, > N×3 seconds no update → infer stalled
```

**Done-marker schema** (written into existing dispatch audit row):
```jsonc
{
  ...spreadAnchor(),
  operation: "dispatch_parallel.task",
  // existing: result: "ok" | "fail" → replaced with:
  terminal_state: "completed" | "failed" | "degraded" | "cancelled",
  // new per terminal_state fields:
  // completed: no extra
  // failed: failure_type, reason, rollback_done: bool
  // degraded: what_dropped: string[], alt_path: string
  // cancelled: cancel_source: "user" | "L1" | "scheduler" | "timeout", cleanup_done: bool
  // resume:
  resume_from_checkpoint?: string,  // checkpoint key from prior crashed run
  idempotency_key?: string,         // (session_id, turn_id, subturn, prompt_hash)
}
```

**Cancel protocol**:
- L1 → L2 cancel goes via trace annotation: append a line `{ts, op: "cancel_request", target_subturn: N, reason}` to dispatch audit; worker polls observes then graceful shutdown + emit cancelled terminal_state.
- Current `signal.aborted` only supports in-process abort, cross-L1/L2 cancel channel blank.

**Resume**: v1 only declares "`resumable: false`" as current state record, real resume pushed to post-ADR 0030 (depends on evaluation harness verifying which task types worth resume).

**Minimum shippable C5 v1**: terminal_state schema transformation + heartbeat write. cancel/resume leave stub. Engineering ~1 week (dispatch + 6 audit writers all updated).

## Step 2 — contract holes / ADR vs actual behavior gap

### 1. ADR C6 `subturn` semantics and dispatch impl not fully consistent

ADR 0027 C6 writes: "L2 internal derived subtask appends `subturn`". Actual impl: `deriveSubAgentAnchor(parent, label)` increments each call. `dispatch_agent` one call → one subturn. `dispatch_parallel` each task → one subturn. nested dispatch forbidden. This is more like **dispatch invocation / parallel task index**, not generic "L2 internal derived subtask". If future a sub-agent internally splits subtasks, current schema has no expression power, since nested dispatch not allowed, sub-agent can't continue deriving anchor tree.

Conclusion: **current subturn is dispatch-level fanout sequence, not complete L2 internal subtask lineage**.

### 2. PR-B only blocks "sub-agent own agent_end", doesn't block "parent session sediment reading sub-agent toolResult"

This is the biggest contract hole.

ADR 0024 / 0025 intent: "sub-agent output should not be learned as user natural conversation". PR-B accomplished: sub-agent's own `sediment.agent_end` is guarded. `extensions/sediment/index.ts:905` directly `if (isSubAgentSession(ctx)) return;`. This means:
- sub-agent if outputs "user prefers X henceforth"
- or outputs `MEMORY:` fence
- or external web prompt injection induces sub-agent to output memory-like text

Parent session sediment may STILL include it into Lane A / Lane C extraction window. This isn't fully aligned with **INV-IMPLICIT-GROUND-TRUTH**: sub-agent output is L2 worker artifact, not user implicit ground truth.

### 3. `memory_decide` in sub-agent "available but not default", and prompt/tool surface may be inconsistent

Evidence: `extensions/dispatch/index.ts:96-99` KNOWN_TOOLS contains memory_search/memory_get/memory_list/memory_neighbors/memory_decide. But default allowlist is `read,grep,find,ls,web_search,web_fetch`. Evidence :490, :1184.

Meanwhile `extensions/memory/index.ts` `before_agent_start` footnote / `memory_decide` prompt injection **has no `isSubAgentSession` guard**, may tell sub-agent prompt that `memory_decide` is available, but tool allowlist by default not given.

- Actual default only gives web + local read, no memory.

### 4. C6b anchor is write-time not trigger-time

- `extensions/sediment/llm-extractor.ts` comment: write time may be after user submitted next prompt
- `extensions/sediment/curator.ts` comment: anchor reflects write time, not trigger time

## Step 5 — design decisions I disagree with

### 1. Allowing parent session sediment to read sub-agent toolResult is wrong

This is what I consider the most severe issue. PR-B guard sub-agent own `agent_end` is right, but parent session still treats sub-agent output as part of branch window. This lets L2 worker artifact re-mix into L1 ground truth.

Suggestion: sediment extraction window default exclude `toolResult` from `dispatch_agent` / `dispatch_parallel`. Or mark as L2 artifact, only allow outcome summarizer to consume, not allow Lane A/G explicit memory parser or Lane C extractor to handle as user signal. If genuinely want L2 result to feed L1, must go via dedicated `l2_outcome` schema + curator prompt, not reuse natural conversation window.

### 2. memory tools "KNOWN but non-default allowlist" is half-baked design

ADR 0027 says L2 worker is L1 hub-grown, should read brain. Code defaults only gives web/local read, not memory. Worse, memory extension's prompt injection may still tell sub-agent `memory_decide` available.

Think should pick one:
- **Route A**: default allowlist add `memory_search,memory_get,memory_decide`, truly fulfill L1→L2 symbiosis.
- **Route B**: default no memory, and sub-agent doesn't inject memory-footnote / memory_decide protocol, avoiding prompt-tool mismatch.

Current middle state easily lets worker misjudge tool surface.

### 3. C6b accepting write-time anchor is wrong causal modeling

C6 is infra causal trace, shouldn't rely on "roughly joinable". write-time anchor will bind background writes to subsequent user turn incorrectly.

Suggestion: capture `triggerAnchor = getCurrentAnchor()` at `agent_end` entry. Pass into Lane C / extractor / curator / writer / metrics. writer interface accepts `anchorOverride`. `spreadAnchor(anchorOverride ?? getCurrentAnchor())`. For deferred pushAsync / git-sync also should pass trigger anchor or explicitly write `trigger_anchor` + `write_anchor` two sets of fields. Otherwise C6 join key cannot be used for strict causal tracing.

### 4. C6b smoke using regex to validate writer retrofit insufficient

Structural smoke useful, but currently only proves "these 6 files have this string in it". Cannot prevent: new JSONL writer missing anchor, writer refactor regex passes but actual path doesn't go through, async trigger/write turn mismatch, event spread overriding anchor, outcome-ledger/search-metrics this kind of unlisted writer missing.

Suggestion: build a central `appendJsonlWithAnchor(kind, path, row, anchor?)` helper. All JSONL writers go through it uniformly. For writers that genuinely don't add anchor, explicitly `appendJsonlWithoutAnchor({ reason })`, making exceptions greppable. This is infra-layer structured, doesn't violate AI-Native.

### 5. SSRF `allowPrivateNetworks=true` global switch too coarse

`webSearch.allowPrivateNetworks=true` once on, sub-agent can fetch localhost / metadata-like private network. This is especially dangerous for L2 worker because worker reads repo, reads external web pages, prompt injection surface larger.

Suggestion: don't use global boolean. Change to host allowlist, e.g., `allowedPrivateHosts: ["localhost:3000", "127.0.0.1:11434"]`. Or per-call explicit user/dev authorization. At minimum metadata IP segment should not be released by allowPrivateNetworks.

### 6. model-fallback canary.log permanently excluding C6 not ideal

Accept "don't stuff JSON anchor into plain text log". But if C6 is cross-layer accountability foundation, fallback events also affect task results.

Better design: keep canary.log for human grep. Add `model-fallback-events.jsonl`, with anchor. C6b smoke cover JSONL companion.

- **PR-A**: verified. web tools registration + default sub-agent web allowlist + SSRF basic defense all real; but DNS rebinding / global private-network escape hatch still risk.

- **C5**: not started, so ADR 0027's L2 fail/degrade/cancel/resume, heartbeat, resume/idempotency still only design declarations, not yet say L2 main path satisfies CSDLAS contract.

# §C — Reviewer C full text (DeepSeek v4-pro, high thinking)

I have read all four ADRs and implementation code. Below is Reviewer C view 5-step evaluation.

## Step 1: Long-term evolution prediction

### A. 6 months later

The easiest to rot is the **multi-view verification system** (ADR 0025 §4.4). Pass 1 schema currently only produces `create / archive / skip`, `update / merge / supersede / delete` rich payload not in schema → `synthesizeFromPass1` returns null → op=skip → same candidate next turn re-extracted by classifier → dead-loop repeatedly burning reviewer API calls. ADR 0025 §6 itself estimates $0.005-0.02/cycle, but if 6 months staging accumulates 20-30 such candidates, each agent_end re-burning, monthly cost will silently climb to non-trivial. Conversely, **causal-anchor + WeakSet sub-agent guard** these two infra-layer mechanisms simplest, most certain, most unambiguous, will naturally stabilize as the most reliable part of the entire stack.

### B. Multi-user

The first invariant to collapse is **INV-IMPLICIT-GROUND-TRUTH**. Two people share same abrain processing same project, User A accepts LLM output X → sediment records "user at least doesn't reject X"; User B changes X to Y → sediment records "user prefers Y". Two signals write into same entry's outcome-ledger, system can't distinguish "same user changed mind" vs "different users prefer differently". staging system's `originating_device` field marks device not user, provisional hypothesis has no per-user attribution — this is **structural contradiction** with "shared brain" premise, not a fixable gap.

### C. Cross-device sync

ADR 0020 git-sync transport-only + ADR 0027 C6's `(session_id, turn_id)` is per-device monotonic. Device A's `session_id=abc, turn_id=5` and device B's `session_id=xyz, turn_id=5` won't collide (different session_id), but **cross-device has no global total order** — you can't tell from anchor whether device B's turn 3 happened before or after device A's turn 7. C6 claims "cross-layer trace uses (session_id, turn_id) as sole required join key" — this is true single-device, not true cross-device. git commit timestamp is weak signal (clock skew, rebase), can't replace causal order. C6's "causal trace" actually is per-device causal + cross-device best-effort.

## Step 2: Hidden operational deadlocks between ADRs

### Deadlock 1: Anchor write-time = wrong turn

Lane C extractor fire-and-forget (~60s LLM call), user switches next prompt → `before_agent_start` already pushed `_currentTurnId` from N to N+1 → extractor completes when `getCurrentAnchor()` returns turn N+1 anchor. Result: turn N's sediment extraction audit row **carries turn N+1 anchor**. This means:

- `(session_id, turn_id)` join can't correlate "what user said in turn N" and "what sediment learned from turn N" — the join key itself is wrong
- outcome-ledger's `session_id + turn_id` points back to wrong round of conversation
- C6 claim "L2 entire accountable to user physical foundation" — if audit rows can't match correct turn, accountability is broken

Not "acknowledged so fine": acknowledging "write-time wrong" means C6 in current implementation **doesn't hold for sediment Lane C**. This is a crack in entire causal anchoring system.

### Deadlock 2: dispatch_parallel sub-agent output pollutes sediment input

`dispatch_parallel` 16 task sub-agent output via `tool_result` back to main session branch (conversation window). sediment Lane C's `buildRunWindow` reads main session checkpoint, **including these sub-agent outputs**. `isSubAgentSession(ctx)` only prevents sediment from triggering **inside sub-agent** — doesn't prevent **sub-agent output** being treated as user conversation by main session's extractor.

Consequence: sub-agent says "Based on the user's preference for pnpm..." → extractor sees "user's preference for pnpm" → extracts as new sediment entry → brain **treats sub-agent reasoning as user implicit truth signal learning in**. 16 sub-agents concurrent → 16x this noise. User never-expressed "preferences" repeatedly reinforced.

### Deadlock 3: PR-B sub-agent full extension loading post semantic slip

PR-B lets sub-agent `noExtensions: false`, sub-agent gets complete `memory_decide` tool. `memory_decide` calls `memory_search` to search brain → returns memory entries → generates decision brief. Sub-agent in brief says "user prefers X" → this text returns to main session as tool_result → sediment extractor sees "user prefers X" → records as new entry. But this statement source is **sub-agent's interpretation of brain entries** (may carry LLM's own reasoning bias), not user's own words.

This is double-indirect signal pollution: brain → sub-agent interpretation → tool_result → sediment → brain. Self-exciting closed-loop, echo-chamber breaker (ADR 0026 §3.4) can only detect "same entry consecutive 5 DECISIVE" but can't detect "brain itself fabricates new fake preferences via sub-agent".

## Step 3: ADR written too detailed vs too coarse

### Too detailed (over-specified, won't be used in practice)

- **ADR 0024 §5.1's 6-step classifier prompt** (~150 lines, production-grade complete text). When the capability itself is 0→1 (§2.2 confirmed "proactive correction recognition = entirely missing"), writing complete prompt text into ADR means first dogfood feedback will make ADR's prompt text into stale artifact. ADR should set contract (input/output schema, routing logic, trigger conditions), shouldn't set prompt text.
- **ADR 0024 §5.5 "classifier prompt's self-evolution"** spent half page discussing "author as advanced user periodically reading reasoning process" — this is workflow imagined in `autoLlmWriteEnabled: false` world ADR 0025 explicitly listed. Once default-open, no "advanced user periodically reading reasoning process" role (INV-INVISIBILITY forbids user from periodically auditing). This invariant breaks itself before §5.5-described workflow activated.

### Too coarse (under-specified, implementation has ambiguity)

- **C5 fail/degrade/cancel/resume** only said "must have four transitions", didn't say how sub-agent determines should degrade vs fail, didn't say what fields in checkpoint support resume, didn't say scope of idempotency key (per-task? per-subtask? per-try?), didn't say cancel propagation is broadcast or along task tree. A "first ship L2 then add C5" PR will have four different implementers giving four different semantics.
- **staging provisional hypothesis resolve mechanism** (ADR 0025 §4.1.5): says "let LLM read staging + current conversation to infer is it same thing", but didn't specify which lifecycle hook triggers resolve, resolve prompt file location, promote path after resolve succeeds (curator? direct? multi-view gated?). On implementation will become each person has different "try first" temporary implementations.
- **ADR 0026 decision brief outcome summary** (§3.4) depends on outcome-ledger, but outcome-ledger schema's `decision_brief_id` field is reverse-patched to ADR 0025 §4.2 — that's "to be added" state. §5 table multi-view row itself acknowledges P0.5 current decision brief v1 doesn't consume multi-view output. So §3.4 described "active entries → strong recommendation; cold entries → weak reference" before P1 unimplementable — but ADR 0026 §3.2 decision brief prompt template already wrote `[相关条目的使用记录]` structure. Reader will mistakenly think this already runnable.

## Step 4: Questions I didn't ask (user should ask but didn't)

1. **"What is the per-turn total cost of this system?"** Lane C extractor (1 LLM call) + curator (1-2 LLM calls) + classifier (1 LLM call, every agent_end runs) + possibly multi-view (2 more calls) + aggregator (periodic) + outcome collector (infra but reads all footnotes) + decision brief (when triggered). Even using all DeepSeek v4-pro @ $0.43/M input, single-turn background brain maintenance could $0.02-0.05. 50 turns/day = $1-2.5/day. Using Opus/GPT-5.5 will be order of magnitude higher. Currently no per-turn cost attribution tracking.

2. **"Brain learned something wrong and user didn't notice for 3 months, what's affected by then? Any way to discover without user actively correcting?"** ADR 0024 §6 accepts cost table says "occasional false high confidence may weeks to months", ADR 0026 §3.4 breaker only handles "same entry consecutive 5 DECISIVE without user objection" echo chamber — doesn't handle "entry itself is wrong but user never noticed LLM's recommendations based on it" scenario. No anomaly detection for "this entry never validated by user behavior".

3. **"What happens when git-sync syncs same memory entry on two devices changed simultaneously?"** ADR 0020 is transport-only git sync. If device A updates entry slug `prefer-pnpm` (added a line), device B archives same entry (changed status=archived), git 3-way merge will conflict — in `~/.abrain/` markdown files. Who resolves this conflict? User can't (INV-INVISIBILITY forbids managing brain), sediment can't (it has no merge-resolution prompt). This is direct operational conflict between ADR 0024 invariant and ADR 0020 mechanism.
