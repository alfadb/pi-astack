# Memory System Vision — 第二大脑记忆系统愿景与设计目标

> 本文记录第二大脑记忆系统的理想形态和设计目标。它是愿景基线，不描述当前实现，不以现有代码或既有 ADR 为约束。

## 0. 反机械主义原则

第二大脑不是条目仓库，也不是状态机。它反对把本该在语境中由认知层重做的判断，冻结为存储层的标量、状态字段或自动推进规则。

- 反机械主义不是反工程结构，而是反结构替代当下判断。索引、字段、分数、流程、分类若被用来决定意义，就越界；若只帮助找到证据、降低成本、保持可追溯，并可被上下文重新解释和越过，则是基础设施。
- 精确与模糊是连续光谱，不是二元分类。状态词只是解释语言，不是判定状态。
- 判断在当下做出，存储只留可重放证据。时间、置信、冲突、当前历史都作为当下判断的输入，而非冻结状态。
- 基础设施可加速召回，但不得代替判断。预过滤只能执行用户以自然语言明确表达的隐私边界这类硬约束，不能自行决定什么重要、什么该被遗忘。
- 已知张力：任何基础设施都可能带来信息损失，例如索引截断、量化、黑箱嵌入。系统应承认这种张力，并在设计上保留回到原始证据链的能力。
- 本节不是合规检查清单。反机械主义是判断原则，不是可自动化验证的规则集；任何试图把本节变成逐条打勾机制的做法，本身就是反机械主义要反对的对象。

## 1. 总体目标

第二大脑是一个长期陪伴用户工作的认知系统。它记住用户经由它处理过的任务、决策、背景、结果、纠正和失败路径，并在后续任务中以合适的方式主动或被动地帮助用户。它不要求用户管理记忆，而是在自然语言交互、真实使用结果和用户纠正中持续演化。

## 2. 记忆对象

- **事件记忆**：记录发生过什么，包括任务、对话、代码修改、调研、决策、失败尝试、用户纠正和后续结果。
- **决策记忆**：记录当时选了什么、为什么选、否决了什么、基于哪些约束、后来是否证明正确。
- **知识记忆**：记录 API、算法、数学约定、工具行为、项目事实、第三方库用法；必须有出处、适用条件、版本、时间和置信度。
- **方法记忆**：记录用户偏好的工作方法、设计原则、沟通方式、取舍倾向；允许模糊、不确定和逐步成型。
- **习惯与协作记忆**：记录用户在不同任务、项目、时间压力和认知负荷下更适合怎样协作；长期人格画像只能作为低置信辅助，不能覆盖当下指令。
- **反向知识**：记录错误判断、走不通的方案、被证伪的假设、导致返工的做法，避免未来重复投入。

## 3. 当前记忆与历史记忆

系统区分当前可用记忆和完整历史。当前可用记忆只保留系统认为现在有价值、应参与召回和决策的内容；完整历史由 Git 版本化保存，用于追溯、解释、恢复和审计。

条目可以从当前记忆中删除，但它的演化路径仍可从 Git 历史中找回。用户心智里不需要归档状态：当前有用就参与当前记忆，当前没用就移出当前视图；历史价值由版本化系统保留。

## 4. 自动演化

所有记忆演化由系统自动完成，包括新增、合并、拆分、更新、删除、跨项目提炼、全局化、降级和恢复。用户不做条目管理，只通过自然语言表达纠正，例如“这个不对”“这只是这个项目的约定”“以后不要这样提醒我”。

系统需要理解纠正对象、作用域、持久度和影响范围，并把纠正传播到相关派生记忆。

## 5. 演化依据

系统不能只因为自己推断就改变记忆。演化应综合多类信号：用户明确表达、用户实际采纳、用户忽略、用户纠正、跨项目重复出现、同类任务复用、后续结果证明、知识来源更新、版本变化、环境变化、项目约束变化。

系统需要区分强信号和弱信号：用户明确纠正通常强于一次行为观察；多项目重复验证强于单项目偶然成功。

## 6. 跨项目与全局提炼

项目知识默认属于具体项目，但系统应能识别可迁移经验。若项目 A 的冷门库 API 用法在项目 B 中出现相同库、相近版本、相似问题和相同调用条件，系统应自然想起并复用，而不是重新调研。

若同一知识在多个项目中被验证，系统可自动提炼为全局知识；若后来发现只在特定版本、框架或项目约束下成立，系统应自动收回全局化结论，改为条件化知识。

## 7. 精确与模糊两类记忆

系统应同时支持两类记忆。第一类是哲学、方法、风格、习惯、人格倾向等软记忆，允许不确定、可变化、可带反例。第二类是 API、算法、数学、项目约定、技术决策等硬记忆，必须精确、有出处、有适用条件、有版本或时间边界。

两类之间可以转化：模糊倾向可能逐渐变成明确偏好，明确结论也可能因反例增加而退回为有条件判断。

## 8. 召回与主动性

系统的目标不是想起越多越好，而是在正确时机想起正确内容。召回需要判断相关性、有效条件、置信度、过期风险、隐私边界和打扰成本。

系统可以主动提醒，但必须克制：当历史知识可能显著减少重复劳动、避免错误、指出冲突或解释背景时主动说；当只是弱相关或可能打断当前任务时保持沉默，除非用户询问。

## 9. 可观察性

用户不管理记忆，但必须能看到系统在运行。界面应展示：刚刚新增、合并、更新、删除了什么；依据是什么；影响哪些项目或全局记忆；哪些内容被提升或降级；哪些高影响变化需要用户注意。

默认展示摘要和高影响变化，必要时可展开到具体来源、diff、时间线和 Git 提交。

## 10. 解释与追溯

用户随时可以问：“你为什么这么认为？”系统必须回答具体依据：来自哪次任务、哪次决策、哪条用户纠正、哪个项目、哪个版本、哪些反例、置信度如何、上次复用结果怎样。

每条当前记忆都应能追溯到历史事件和演化过程；每次演化都应能说明原因。

## 11. 隐私与边界

不是所有记忆都能跨项目或全局化。系统必须识别密钥、客户信息、私有业务逻辑、公司机密、临时情绪表达和仅限项目的约定。高复用价值不能压过隐私边界。

某些内容只能在原项目内召回，某些内容只能以脱敏形式参与全局提炼，某些内容不应沉淀为长期记忆。

## 12. 反对与校正能力

系统不应只学习如何顺从用户。若当前选择与过去明确决策、已验证事实、项目约定或失败路径冲突，它应主动指出，并说明依据和不确定性。

它可以配合用户快速推进，也应在证据足够时提出反对意见。目标不是让用户少纠正，而是让用户更少重复错误、更快获得可靠判断。

## 13. Git 与持久底座

Git 是持久记忆的默认底座，承担完整历史、非敏感证据与演化路径的存储。当前记忆由工作树或索引视图表达；完整历史由 Git 提交表达。语义判断、有效条件、召回时机、跨项目提炼、自然语言纠错仍由第二大脑的认知层完成。

敏感内容若必须持久化，遵循与 Git 同构的版本化、diff、可追溯原则，但物理隔离于独立存储边界。隐私边界由自然语言描述，不由权限等级或存储格式决定；系统不将“必须入 Git”作为机械规则，而是将“版本化、diff、可追溯”作为组织原则。

Git 负责保存演化路径、支持删除后的恢复、展示 diff、处理跨设备同步和追溯内容变更来源；它不是理解系统，也不替代认知层的判断。

## 14. 一句话版本

理想第二大脑不是一个条目仓库，而是一个自动演化的个人认知系统：它用 Git 保留完整历史，用当前视图服务当下任务，用自然语言接受纠正，用真实结果持续改进，并在跨项目、跨会话、跨时间的背景中帮助用户记住、判断、复用、反省和避免重复错误。

## 附录：用语约定

以下用词仅为减少沟通歧义的解释性说明，不构成判定规则或检查清单；任何试图将其作为机械合规依据的做法，本身就是反机械主义所要反对的。

- **认知层**：做出语义判断、召回时机、跨项目提炼、自然语言纠错的主体，不是具体模块名。
- **存储层**：保存证据、支持版本化与检索的物理或逻辑基础设施，不替代认知层判断。
- **标量 / 状态字段**：指任何试图用单一数值或离散标签冻结语义判断的存储形式，如固定分数、刚性分类、布尔状态。
- **默认底座**：指非敏感内容的标准持久化方式，不排除因隐私边界而物理隔离的例外。
- **自然语言隐私边界**：由用户以自然语言明确表达的约束，如“这是客户机密”“仅限该项目”，系统据此判断，而非依赖预设的权限等级或存储格式。
- **可重放证据**：足以让认知层在需要时重新做出判断的原始材料，不一定是完整历史记录。
